# 第 7 章 前端实战，从页面需求到可运行原型

## 本章目标

通过一个完整前端案例，让读者看到 Codex 如何参与页面交付全过程。

## 你将学会什么

- 如何拆解前端页面需求
- 如何让 Codex 在已有风格内完成界面修改
- 如何逐步推进而不是一次性梭哈
- 如何把前端需求变成一轮轮可验证的小任务

## 为什么前端是最适合练 Codex 的实战场景之一

前端实战特别适合拿来练 Codex，原因很简单：它足够具体，又足够常见。

一个典型前端需求通常同时包含：

- 结构搭建
- 样式约束
- 状态处理
- 数据请求
- 交互反馈
- 验收标准

也就是说，它既不是简单到一句话能做完，也没有复杂到必须一开始就拉满全量上下文。只要你会拆，它非常适合做一轮完整的 AI 协作演练。

不过，前端也是最容易让人误用 Codex 的场景。因为页面结果可视化很强，新手很容易一上来就说“帮我做个页面”，然后期待它像魔法一样一步到位。结果通常是做得很热闹，但很难真正接入现有项目。

所以这一章的重点不是“让 Codex 自由发挥设计”，而是让它在明确限制下参与交付。

## 以一个真实需求为例

我们假设现在有一个常见的后台系统需求：

“做一个用户列表页，支持关键词搜索、状态筛选、表格展示和分页。”

这个需求看起来很普通，但已经足够包含一个完整前端交付链条：

- 页面的信息结构
- 搜索和筛选区域
- 表格字段展示
- 分页和请求参数联动
- 空态和加载状态

如果你直接把这句话丢给 Codex，大概率会得到一个通用程度很高、和你项目风格不太匹配的结果。更好的做法，是把它拆成多个阶段。

## 第一步：先拆页面，不急着写代码

前端需求最怕的不是写不出来，而是方向一开始就偏了。

所以第一轮，你应该先让 Codex 帮你做页面任务拆解，而不是直接生成整页代码。

参考提示词：

```text
我需要做一个后台用户列表页，包含关键词搜索、状态筛选、表格和分页。
请先不要直接写代码，先帮我完成以下事情：
1. 拆解这个页面的主要区域
2. 说明每个区域要承载什么信息
3. 给出建议的实现顺序

约束：
- 这是后台页面，不是营销页
- 以实用和清晰为主
- 先给结构方案，不做视觉发挥
```

这一轮的目的，是先把“页面长什么样”和“开发顺序怎么排”确定下来。

## 第二步：锁定现有风格，不让它胡乱发挥

前端项目里，一个很现实的问题是：**你不是在凭空做页面，而是在已有设计系统和代码风格中做页面。**

如果你不把这件事说清楚，Codex 很容易给你一个能看但不适配的结果。

所以你需要显式补充约束，例如：

- 保持现有组件库用法
- 保持已有表格和表单风格
- 不新引入设计语言
- 优先复用当前页面布局方式

这一步非常重要。前端很多返工，都是因为第一轮没有把风格边界锁住。

## 第三步：按“结构 -> 数据 -> 交互”推进

不要一上来让 Codex 同时搞定一切。更稳的顺序通常是：

### 1. 先做结构

先确认页面区域和基础骨架：

- 搜索区域放哪里
- 筛选项怎么布局
- 表格列有哪些
- 分页区域如何放置

这一轮重点是页面骨架，不必急着打通数据。

### 2. 再做数据

结构确认后，再让 Codex 协助你思考：

- 请求参数有哪些
- 搜索和筛选如何映射到接口参数
- 分页状态如何管理
- 首次加载和查询重置逻辑怎么处理

### 3. 最后做交互

到这一步，再补这些细节：

- 查询按钮行为
- 重置按钮行为
- 加载态和空态
- 分页切换与数据刷新

这种推进顺序，会显著降低一口气写废的概率。

## 一个更实用的前端协作三轮法

### 第一轮：分析和拆解

```text
我需要实现一个后台列表页，包含关键词搜索、状态筛选、表格和分页。
请先帮我：
1. 拆解页面区域
2. 列出建议的组件结构
3. 说明实现顺序

要求：
- 保持后台系统风格
- 不先写完整代码
```

### 第二轮：生成骨架

```text
基于上面的结构，请生成页面骨架代码。

要求：
- 保持现有项目风格
- 先搭结构和占位，不急着补完所有逻辑
- 不改无关文件
```

### 第三轮：补逻辑和验收

```text
现在请在已有骨架基础上，补齐搜索、筛选和分页的核心逻辑。

要求：
- 只做与当前页面相关的最小改动
- 说明涉及哪些状态和请求参数
- 最后补一个验收清单
```

这三轮法的本质，是把前端需求从“整页创作”变成“阶段性交付”。

## 前端实战里最常见的几个坑

### 坑一：不给设计约束

你不说要延续现有风格，它就会默认自己发挥。最后也许做得不丑，但一定不统一。

### 坑二：把结构、逻辑、交互一次性打包

这样会让结果非常难检查。你会同时面对布局问题、状态问题、参数问题和交互问题，调试成本极高。

### 坑三：忽略接口和状态管理

很多新手看前端，只看页面长什么样，却忘了真正难的是数据流。Codex 也一样，如果你只盯着视觉效果，它很容易把数据侧做得很粗糙。

### 坑四：没有验收清单

前端需求如果没有明确验收项，就很容易陷入“看着差不多”。你应该明确：

- 搜索是否生效
- 筛选是否与参数联动
- 分页切换是否正确刷新
- 空态和加载态是否合理

## 验收意识比生成速度更重要

很多人使用 Codex 时，会特别关注它“写得快不快”。这是一个容易让人走偏的标准。

在真实项目里，更重要的是：

- 写出来的东西能不能接入现有代码
- 改动范围是否可控
- 验收路径是否清晰

如果它写得很快，但你需要花更多时间返工、改风格、补数据流，那总体效率并不高。

所以前端实战最成熟的用法，不是让 Codex 快速铺满页面，而是让它在你设定好的轨道内持续推进。

## 本章总结

前端实战的关键不是让 AI “自由设计”，而是把它锁定在明确结构、明确风格、明确验收的框架里。

你应该带走这些结论：

- 前端需求先拆，不要直接整页开写
- 风格约束必须明确说出来
- 最稳的推进顺序是“结构 -> 数据 -> 交互”
- 验收清单决定这次协作是不是交付而不是演示

只要你能把一个列表页场景跑顺，后面的表单页、详情页、弹窗页，都会越来越容易。

## 实战任务

选一个你手头真实存在的前端页面需求，至少按三轮法完成一次协作：

1. 让 Codex 拆页面结构
2. 让它生成页面骨架
3. 让它补逻辑并给出验收清单

在这三轮里，强制自己每轮只解决一类问题，不要贪快。
