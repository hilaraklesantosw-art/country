# 第 6 章 文件阅读、代码修改、调试和测试协作

## 本章目标

帮助读者把 Codex 纳入完整开发闭环，而不是停留在“帮我解释一下”的阶段。

## 你将学会什么

- 如何围绕文件组织任务
- 如何让 Codex 参与调试和排错
- 如何结合测试验证结果
- 如何把“分析 -> 修改 -> 验证”做成闭环

## 真正能拉开差距的，不是会问，而是会闭环

很多人对 Codex 的使用，停留在“它能不能给我一个像样的答案”。这还远远不够。

在真实开发里，真正有价值的不是某个漂亮回答，而是从问题出现到结果验证的完整闭环。也就是：

1. 理解当前对象
2. 定位问题范围
3. 提出最小修改
4. 验证修改是否有效
5. 根据结果继续迭代

如果你只有第一步和第二步，没有后面的验证和复盘，那么 Codex 的价值就很难稳定落地。

## 为什么“文件”是最好的协作单位

很多新手喜欢直接从“大需求”开始和 Codex 协作，比如“把这个功能做好”“把这里优化一下”。问题是，大需求往往横跨多个模块，信息过于分散。

一个更稳妥的方式，是尽量把任务收敛到“文件级”或“对象级”。

例如：

- 这个页面文件负责什么
- 这个服务文件在哪里处理请求
- 这个组件为什么渲染异常
- 这个测试文件为什么断了

文件是非常好的协作单位，因为它具备三个优点：

- 边界天然清晰
- 容易验证
- 便于逐步扩大范围

所以当你遇到问题时，不要一开始就问“整个项目哪里有问题”，而是先努力缩到几个具体文件。

## 报错不是噪音，而是排查入口

新手一看到报错，常常会慌。要么直接复制整段报错给 Codex，希望它“瞬间看穿”；要么完全忽略报错，只说“项目跑不起来了”。

这两种方式都不够好。

报错最大的价值，不是它能直接告诉你答案，而是它能帮助你缩小范围。

你应该学会把报错信息转成结构化排查输入。

例如：

- 报错发生在什么命令下
- 报错指向哪个文件、行号或模块
- 是运行时报错、构建时报错还是测试报错
- 最近刚改过什么

当你把这些信息整理清楚后，Codex 就更容易参与排查，而不是泛泛给建议。

## 一个典型调试场景

假设你执行前端构建时看到报错，页面白屏，控制台提示某个模块导入失败。

糟糕问法：

```text
项目炸了，帮我看看。
```

这种提问没有任何工程信息。

更有效的问法：

```text
我在执行前端构建时遇到报错，请先帮我分析原因，不要直接给大改方案。

现象：
- 构建失败
- 控制台提示某模块导入失败

上下文：
- 最近刚改过用户列表页相关代码
- 报错指向 [这里写文件或模块]

请你完成：
1. 列出最可能的 3 个原因
2. 说明每个原因应该先检查什么
3. 给出最小验证步骤
```

这类问题一旦结构化，Codex 的价值就会明显上升。

## 分析、修改、验证，顺序不能乱

很多人失败，是因为顺序反了。

一看到问题，就立刻让 Codex 改代码。这样做的问题是，你自己还没确认问题范围，Codex 也没明确验证路径，最后可能改了一堆地方，问题还不一定解决。

更稳的顺序应该是：

### 第一步：先分析

先让它做这些事：

- 总结现象
- 猜测高概率原因
- 指出最相关文件
- 给出验证优先级

### 第二步：再修改

只有在分析足够清楚后，再要求它：

- 给出最小改动方案
- 说明为什么要这么改
- 标出可能受影响的地方

### 第三步：最后验证

不要停在“代码看起来有道理”。你应该继续要求：

- 给出验证步骤
- 说明需要跑哪些测试
- 列出哪些行为应该重点检查

这一套流程看似慢，实际上比盲改快得多。

## 测试不是附属品，而是协作终点

很多人在和 Codex 协作时，会自然把重点放在“改出来”，却忽略了“如何证明改对了”。

这就是为什么测试非常重要。

测试在这里有两层价值：

- 第一，验证修改是否真的解决问题
- 第二，防止新改动引入副作用

即使你所在项目测试覆盖不完整，你也至少要养成让 Codex 给出验证思路的习惯。比如：

- 应该重点测哪些路径
- 哪些边界情况容易漏
- 如果没有现成测试，最小人工验证是什么

只要你开始这样做，使用质量就会明显上一个台阶。

## 一个更成熟的协作流程示例

下面给你一个真实开发里非常好用的三轮法。

### 第一轮：理解对象

```text
请先解释这个文件的职责，不要修改代码。
我想知道：
1. 它主要负责什么
2. 它依赖了哪些关键模块
3. 哪些地方最可能和当前问题有关
```

### 第二轮：定位问题

```text
我现在遇到的问题是：[问题现象]
请结合刚才分析的文件，帮我：
1. 列出最可能的原因
2. 给出优先排查顺序
3. 说明每一步验证方式
```

### 第三轮：提出修改与验证

```text
基于以上分析，请给出最小修改建议。

要求：
- 不改无关逻辑
- 保持现有风格
- 同时给出修改后的验证步骤
```

这种三轮法的好处是，你不会在还没想清楚时就进入代码修改。

## 使用 Codex 调试时最常见的错误

### 错误一：只贴报错，不贴上下文

报错只是入口，不是全部。你还需要告诉它：

- 这是在哪个阶段发生的
- 最近改了什么
- 哪个模块最可疑

### 错误二：一上来就让它重构

大多数问题只需要定位和最小修改，不需要大面积调整。

### 错误三：修改后不做验证

这是最危险的一点。代码写出来不等于问题解决。你必须验证。

### 错误四：忽视失败结果的价值

如果某个验证没通过，不代表这轮协作失败。恰恰相反，失败结果本身也是新上下文，可以继续拿给 Codex 缩小范围。

## 本章总结

真正高效的使用方式，不是“让 Codex 一次写完”，而是让它加入可靠闭环。

你应该带走这些结论：

- 文件是非常好的协作单位
- 报错要结构化，不要情绪化
- 先分析，再修改，最后验证
- 测试和验证步骤是协作终点，不是附属品
- 失败结果也是有价值的新上下文

当你开始这样使用 Codex，它就不再只是解释器，而会逐渐成为调试和交付助手。

## 实战任务

拿一个真实报错，完成下面这条完整链路：

1. 让 Codex 解释相关文件职责
2. 让它列出最可能的原因
3. 让它给出最小修改建议
4. 让它补上验证步骤
5. 把验证结果再反馈给它做一次复盘

只要你能走完这一轮，你的使用水平就已经超过只会“随便问问”的阶段了。
