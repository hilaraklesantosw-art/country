# 第 3 章 第一次上手，界面、基本操作、最小可用提示词

## 本章目标

让读者完成第一次真正可用的 Codex 操作，摆脱面对输入框不知道说什么的状态。

## 你将学会什么

- 如何发出第一个有效任务
- 如何组织一次最小可用请求
- 如何判断 Codex 的第一轮输出是否合格
- 如何从“乱问”切换到“有目的地问”

## 为什么很多人的第一次上手体验很差

第一次打开 Codex，很多人都会陷入一种尴尬状态：工具已经摆在面前，但脑子里只剩下一句空话，“我现在要说什么？”

这不是能力问题，而是任务表达问题。

很多新手第一次用 Codex 时，会走向两个极端。第一种是过度简化，只输入一句“帮我看看这个项目”或“帮我写代码”。第二种是把所有想法一股脑倒进去，写成一大段背景、顾虑、猜测和需求混在一起的长消息。前者信息不足，后者重点模糊，结果都很容易不稳定。

所以第一次上手，不要追求复杂，也不要追求高级。你真正要做的，是完成一次最小但有效的成功体验。只要这次体验是清晰的、可验证的、可复用的，后面你就会越来越顺。

## 第一次上手，不要做太大的任务

第一次就让 Codex “帮我做完整系统”“帮我重构整个项目”“帮我一次性修完所有问题”，这几乎是在强行制造失望。

原因很简单。大任务不仅需要大量上下文，还需要你对业务边界、风险、优先级有清晰理解。很多时候，这连人类工程师之间都要反复沟通，更别说 AI 工具。

第一次上手更适合做这些任务：

- 解释一个项目的结构
- 找出入口文件和核心目录
- 解释某个文件或函数在做什么
- 读一条报错并推测排查方向
- 给出某个修改的第一步建议

这些任务的共同特点是：

- 范围小
- 容易验证
- 不容易误伤项目
- 能快速感受到 Codex 的真实价值

## 最小可用提示词公式

如果你现在什么都不想记，只记住下面这个结构就够了：

**任务 + 上下文 + 约束 + 期望输出**

这不是唯一写法，但对新手非常实用。

### 1. 任务

你到底想让 Codex 做什么。

不要说：

- 帮我看看
- 你觉得这个怎么样
- 帮我处理一下

这种表达在工程里没有执行价值。更好的方式是直接写动作：

- 帮我总结项目结构
- 帮我解释这个函数的作用
- 帮我找出这个报错最可能关联的文件
- 帮我给出最小修改建议

### 2. 上下文

它需要基于什么信息来完成任务。

上下文可能包括：

- 当前项目或代码文件
- 相关功能模块
- 报错信息
- 当前所处阶段
- 你已经知道的线索

很多人提问失败的根本原因不是“不会问”，而是默认 AI 能自动补齐上下文。它不会。

### 3. 约束

你希望它遵守什么边界。

这一步非常重要，因为它可以防止 Codex 过度发挥。最常见的约束有：

- 先分析，不要修改代码
- 只做最小建议
- 不要改无关文件
- 保持现有风格
- 不确定的地方请标出来

### 4. 期望输出

你希望它怎么回答。

例如：

- 用列表回答
- 先给结论，再说明理由
- 只列最关键的 3 个文件
- 按“问题/原因/建议”输出

这一步的价值在于提高可读性和可复用性。

## 一个典型对比：差提示词和好提示词

假设你接手一个陌生仓库，想快速理解它。

差提示词：

```text
帮我看看这个项目。
```

问题很明显：

- 没说你要知道什么
- 没说当前是分析还是修改
- 没说你希望答案长什么样
- 没说关注哪个方向

更好的写法：

```text
请先帮我理解这个项目，不要修改代码。
我想知道：
1. 项目主要是做什么的
2. 入口文件在哪里
3. 哪几个目录最关键
4. 如果我要改用户管理功能，建议先看哪些文件
请用简短列表输出。
```

这段话为什么有效？因为它具备了完整最小结构：

- 任务：理解项目
- 上下文：当前仓库，重点关注用户管理
- 约束：不要改代码
- 输出：简短列表

这就是你第一次上手最该模仿的思路。

## 第一次上手最推荐做的 3 个动作

### 动作一：让 Codex 总结仓库结构

这是最稳妥的开局方式。

参考提示词：

```text
请先帮我理解这个项目，不要修改代码。
我希望你说明：
1. 项目的主要用途
2. 入口文件或核心启动位置
3. 最关键的 3 到 5 个目录
请尽量基于实际文件回答，不要泛泛而谈。
```

这一步会让你立刻知道，它在“读结构”和“做第一轮整理”上是否能帮到你。

### 动作二：让 Codex 解释单个文件或函数

结构理解之后，就该缩小范围。

你可以继续问：

```text
请解释这个文件的主要职责。
我更关心：
1. 它解决什么问题
2. 它依赖了哪些关键模块
3. 如果我要修改它，最容易出错的地方是什么
请用通俗语言说明。
```

这一轮的重点不是求代码，而是求理解。只要你能快速搞懂一个原本陌生的文件，第一次使用就已经值回来了。

### 动作三：让 Codex 给“第一步建议”

注意，不是让它直接开写完整代码，而是让它告诉你“第一步该怎么走”。

例如：

```text
我接下来想修改用户列表页的筛选逻辑。
请不要直接写完整代码，先告诉我：
1. 应该优先查看哪些文件
2. 哪些状态、接口或组件可能受影响
3. 第一轮最小改动应该从哪里开始
```

这个习惯很重要。很多人失败，不是因为 Codex 不会写，而是因为自己太快把它推进了高风险修改阶段。

## 如何判断第一轮输出是否合格

新手最容易犯的另一个错，是把“语气像对的”误当成“工程上可用”。

你至少应该用以下四个标准检查第一轮输出。

### 1. 它有没有真的回答你的问题

你问的是“入口文件在哪”，它却开始讲框架概念和一般性知识，这就是跑偏。

### 2. 它有没有基于实际上下文回答

如果它的回答里几乎没有具体文件、目录、函数名，只有泛泛总结，那说明它可能没有真正吃到你给的上下文。

### 3. 它有没有擅自扩张任务

你让它先分析，它却开始建议大规模重构；你让它解释原因，它却直接给重写方案。这不一定是聪明，常常是偏题。

### 4. 结论是否容易验证

第一次上手最好选择那种可以立刻核对的输出。比如：

- 它说某个文件是入口，你能马上打开验证
- 它说某个模块负责状态管理，你能在代码里确认
- 它说某个函数负责请求封装，你能快速搜索到调用关系

可验证，才会积累信心和经验。

## 新手第一次上手最常见的错误

### 错误一：问题太大

“帮我做一个项目。”

这类问题没有边界，也没有阶段，几乎注定得不到稳定结果。

### 错误二：问题太空

“这个怎么弄？”

你脑子里也许有场景，但你没说出来。AI 不会读心。

### 错误三：不给边界

你没有说“先分析”，也没有说“不要修改无关文件”，结果它就会按自己的默认逻辑补全任务。

### 错误四：看见答案流畅就停止思考

这是最危险的错误。AI 最大的迷惑性不在于乱说，而在于有时会把错误说得很顺。工程判断不能停在“读起来像那么回事”。

## 一套可以直接复制的首次上手模板

如果你不想自己组织，可以直接用下面这套：

```text
请先帮我理解当前项目或文件，不要急着修改。
我的目标是：[这里写你的目标]

请你完成：
1. 说明当前对象主要负责什么
2. 找出与目标最相关的文件、函数或模块
3. 提醒我最容易忽略的风险点
4. 给出下一步最小动作建议

要求：
- 先分析，不直接大改
- 基于实际内容回答
- 不确定的地方明确说明
- 用简短列表输出
```

这套模板不华丽，但非常实用。对新手来说，稳定比花哨更重要。

## 本章总结

第一次上手的关键，不是让 Codex 显得多强，而是让你掌握一种可复用的基本提问方式。

你至少应该记住三件事：

- 第一次任务要小、清楚、可验证
- 提示词最小结构是“任务 + 上下文 + 约束 + 期望输出”
- 先分析，再修改，比直接让它开写更稳

只要你把这三点练熟，后面不管是读代码、修 Bug、做页面还是写接口，都会顺很多。

## 实战任务

选择一个真实仓库或真实文件，向 Codex 发出一条请求，要求它：

1. 解释当前对象的主要职责
2. 指出最相关的 3 个文件或模块
3. 给出下一步最小动作建议

完成后，自己复盘两件事：

- 哪一句提示词最有效
- 哪一句描述不够清楚，导致它回答偏了

这份复盘后面会越来越值钱。
