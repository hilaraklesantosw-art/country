# 第 5 章 从问答到交付，常见开发任务模板

## 本章目标

让读者掌握几个可以反复复用的任务模板，把聊天行为升级为交付行为。

## 你将学会什么

- 如何把日常开发需求写成模板
- 如何让 Codex 更稳定地产出可用结果
- 如何围绕模板形成个人工作流
- 为什么“模板化”是稳定提效的关键

## 为什么很多人一直停留在“问答层”

很多人用 Codex 的方式，本质上还是在聊天。

他们会：

- 问一个知识点
- 让它解释一段代码
- 让它给个思路
- 让它优化一小段文字

这些都不是没价值，但它们很难稳定转化为交付。真正有差距的人，不是问得多，而是能把常见任务沉淀成模板，让 Codex 反复参与交付。

所谓“从问答到交付”，并不是要求你一次性把任务全交给 AI，而是把高频、重复、结构相对固定的任务，变成一组可复用的协作模式。

## 模板化的真正价值

模板不是偷懒，而是把好用的方法固定下来。

当你不做模板时，每次任务都像重新发明轮子。你今天靠灵感写了一段不错的提示词，明天又忘了；今天一个测试补得很好，后天同样场景却问废了。问题不在于 Codex 不稳定，而在于你自己的输入结构不稳定。

模板化至少能带来四个收益：

- 降低提问成本
- 稳定输出质量
- 方便复盘和迭代
- 帮你形成个人工作流资产

你可以把模板理解为“已经验证过有效的协作起点”。

## 哪些任务最适合模板化

并不是所有任务都值得做模板。最适合模板化的，通常是那些：

- 高频出现
- 结构相似
- 需要稳定输出
- 对上下文要求可控

开发里最典型的就是以下几类。

### 1. 读代码模板

场景非常常见：

- 接手陌生项目
- 理解旧模块
- 追某条调用链
- 搞清某个文件职责

这类任务很适合做模板，因为你每次关心的问题都差不多：

- 它是做什么的
- 关键入口在哪
- 依赖关系是什么
- 如果要改，风险点是什么

参考模板：

```text
请先帮我理解当前代码对象，不要修改代码。

当前对象：
[文件/函数/模块]

请你完成：
1. 说明它的主要职责
2. 找出它依赖的关键对象
3. 说明如果我要修改它，最容易影响到哪里

输出要求：
- 用简短列表回答
- 先结论后说明
- 不确定的地方明确标注
```

### 2. 写函数模板

很多函数都不是从零创新，而是在已有项目约束下写一个清晰、可运行、符合风格的实现。

这类任务的关键不是“帮我写代码”，而是把边界讲清楚。

参考模板：

```text
我需要新增一个函数。

目标：
[函数要解决的问题]

上下文：
[所在文件/模块/已有相关代码]

约束：
- 保持现有代码风格
- 尽量复用已有工具函数
- 不引入新依赖

请你完成：
1. 先说明实现思路
2. 再给出函数代码
3. 提醒我需要特别验证的边界情况
```

### 3. 修 Bug 模板

这是特别值得模板化的一类。

很多人修 Bug 时会对 Codex 说一句“帮我看看这个报错”，然后收获一堆泛泛分析。真正有效的做法，是把 Bug 排查过程结构化。

参考模板：

```text
我正在排查一个问题，请先分析，不要直接给大改方案。

问题现象：
[报错信息/异常表现]

相关上下文：
[相关文件/调用位置/最近改动]

请你完成：
1. 列出最可能的 3 个原因
2. 说明每个原因对应应该先检查什么
3. 给出最小验证步骤

输出要求：
- 按“原因 -> 检查点 -> 验证方式”输出
- 不要跳过推理过程
```

### 4. 写测试模板

很多人知道测试重要，但又懒得每次从零组织。Codex 在这里特别适合做提效助手。

参考模板：

```text
我需要为现有逻辑补测试。

当前对象：
[函数/接口/组件]

请你完成：
1. 说明最值得覆盖的行为分支
2. 给出测试用例清单
3. 按现有测试风格补一版初稿

约束：
- 保持当前测试框架和风格
- 不修改业务逻辑
- 优先覆盖核心路径和边界情况
```

### 5. 改文档模板

很多交付最终卡的不是代码，而是说明不清。

例如：

- 需求说明混乱
- 改动说明太散
- 验收标准不明确
- 交接文档没人看得懂

这些都很适合模板化。

参考模板：

```text
请帮我把下面的内容整理成更清晰的开发说明。

目标读者：
[同事/测试/产品/客户]

原始内容：
[原文]

请你完成：
1. 整理结构
2. 提炼关键结论
3. 补出容易遗漏的注意事项

输出要求：
- 用标题和列表组织
- 语言直接，不写空话
```

## 为什么模板能显著提高结果稳定性

你可能会问：同样是提问，模板真的有这么重要吗？

答案是，有。

因为绝大多数不稳定，不是模型随机，而是你每次输入结构都在变。

今天你先说背景，明天你先说结果；今天你要求列表，明天你什么也没说；今天你加了“先分析”，明天你又忘了。每次输入都不同，输出当然也容易波动。

模板的作用，就是把那些已经验证有效的结构固化下来。它不保证永远完美，但会显著减少无谓波动。

## 让模板真正有用的三个原则

### 1. 模板必须服务于具体任务

不要写“万能模板”。越想万能，越容易变空。真正好用的模板，几乎总是针对具体任务场景。

### 2. 模板必须允许填空

模板不是死板文本，而是结构。你要能替换：

- 当前对象
- 当前目标
- 当前限制
- 当前输出要求

### 3. 模板必须经过复盘

第一次写出来的模板，通常不会是最优的。真正值钱的是你连续使用三五次后，知道哪些句子最关键，哪些句子是废话，然后不断收紧。

## 一个现实建议：先做 3 个自己的高频模板

如果你现在刚开始系统化使用 Codex，不要贪多。先做 3 个模板就够了：

1. 读代码模板
2. 修 Bug 模板
3. 写测试或改文档模板

只要你把这 3 个模板跑顺，后面的模板会自然长出来。

## 本章总结

从问答到交付，关键不是让 Codex 一次做更多，而是让你把常见任务变成可复用结构。

你应该带走这些结论：

- 模板化是稳定提效的核心
- 最适合模板化的是高频、重复、边界相对清晰的任务
- 模板不是死文案，而是固定协作结构
- 真正值钱的模板，是被你反复验证和迭代过的模板

一旦你开始积累模板，Codex 就不再只是一个聊天窗口，而会逐渐变成你的工作流组件。

## 实战任务

现在请你为自己常做的三类任务，各写一个模板。建议优先写：

1. 读代码模板
2. 修 Bug 模板
3. 写测试或改文档模板

然后把它们放到你自己的笔记里，接下来一周只用这三套结构和 Codex 协作，再回来改进。
