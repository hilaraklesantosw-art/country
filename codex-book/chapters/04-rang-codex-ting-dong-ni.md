# 第 4 章 让 Codex 听懂你，任务描述、上下文、约束条件

## 本章目标

帮助读者从“会问”升级到“问得准”，显著提高输出稳定性。

## 你将学会什么

- 如何描述任务目标
- 如何给足上下文而不过载
- 如何用约束条件避免跑偏
- 如何通过迭代澄清把模糊需求变成可执行任务

## 从“能问”到“问得准”，中间差的是什么

如果说上一章解决的是“我终于敢开口问了”，那这一章解决的是另一个更现实的问题：为什么我已经会提问了，结果还是时好时坏？

答案通常不是模型能力，而是任务表达方式。

很多人已经意识到不能只说“帮我看看”，于是开始写更长的提示词。但长，不代表准。你可能说了很多背景，却没有讲清楚这次到底要完成什么；你也可能写了很多细节，却没有指出什么是不能破坏的边界。结果就是，Codex 收到一大堆信息，却不知道该把注意力放在哪里。

所以真正的升级，不是把提示词写得越来越长，而是写得越来越有判断力。

## 一个高质量任务描述通常包含四层信息

你可以把稳定的任务描述拆成四层：

1. 目标是什么
2. 当前对象是什么
3. 什么不能破坏
4. 结果应该长什么样

这四层里，最容易被忽略的是第三层：**什么不能破坏。**

很多失败不是因为 Codex 不努力，而是因为你没告诉它哪些地方不能动。比如：

- 不要修改接口结构
- 不要引入新依赖
- 保持当前页面风格
- 只做最小改动
- 先分析原因，不直接修

这些信息往往比“请帮我实现功能”更重要。

## 先说目标，再给背景

很多人写任务时有个习惯：一上来先铺陈大量背景，讲团队历史、项目旧债、需求由来、自己的担忧，结果说了半天还没说到这次到底要干什么。

这样做的问题是，重点会被稀释。

更高效的顺序通常是：

1. 先说这次要完成什么
2. 再补完成这件事所需的背景
3. 最后补上约束和输出要求

例如，低效写法：

```text
我们这个项目之前是别人做的，结构比较老，也挺乱的。最近产品提了个需求，想在用户列表这里加一个状态筛选。我看了一下，好像和表单、接口参数、页面状态都有关，但我不太确定要从哪里下手。你帮我看一下。
```

更高效的写法：

```text
我需要给用户列表页增加“按状态筛选”功能。
请先不要直接写完整代码，先帮我完成三件事：
1. 找出最相关的文件
2. 说明请求参数和页面状态可能在哪里处理
3. 给出第一轮最小修改建议

约束：
- 保持现有代码风格
- 不引入新依赖
- 不确定的地方先标出来
```

后者并没有更长，但更容易执行。

## 上下文不是越多越好，而是越相关越好

一听到“要给足上下文”，很多人就会进入另一个极端：那我是不是应该把一切都塞进去？

不需要。

上下文的作用，是帮助 Codex 缩小理解范围，而不是把它淹没在无关信息里。

最值得优先提供的上下文通常包括：

- 当前任务直接相关的文件、函数、页面、接口
- 你现在所处的阶段：分析、定位、设计、修改、测试、复盘
- 不能破坏的边界
- 你的关注重点：兼容性、速度、最小改动或可维护性

如果你给了很多背景，却没有这些核心信息，输出依然可能不稳定。

## 不要把模糊需求原样丢给 Codex

真实工作里，需求本来就经常模糊。比如：

- “这个页面不够清楚，优化一下”
- “这里体验不太好，调整一下”
- “把导出功能补一下”

如果你把这种模糊需求原样扔给 Codex，它大概率也会模糊地回你。因为你自己都还没完成需求翻译。

你应该先做一步人工加工，把模糊表述转成工程任务：

- 优先优化什么
- 哪些地方不能动
- 成功标准是什么
- 本轮要先分析还是直接实现

这说明一件很重要的事：高质量提示词背后，往往不是语言技巧，而是任务思考能力。

## 约束条件是防止跑偏的刹车系统

很多人担心加约束会限制 AI 发挥。实际上，约束通常是在帮助它更快进入你的真实目标。

你可以把约束理解为刹车系统。不是为了让车跑不动，而是为了让它不要冲出赛道。

开发任务里最常见的约束有：

- 先分析，不直接修改
- 只做最小改动
- 不改无关文件
- 保持现有代码风格
- 不引入新依赖
- 不确定的地方明确说明
- 先给方案，再给代码

当你养成写约束的习惯后，会明显发现输出更稳。

## 输出要求直接影响你的处理成本

很多人认真写了任务和背景，却忘了告诉 Codex 应该怎么回答。结果它给你一大段看似丰富、实则难以消费的内容。

所以输出要求不是细枝末节，而是效率开关。

你可以要求它：

- 用列表回答
- 先结论后理由
- 只列文件名和作用
- 按“问题/原因/建议”输出
- 按“假设/验证步骤/可能改动”输出

输出结构一旦稳定，后续复用和比较都会容易得多。

## 一个前端案例：同一个需求，为什么结果差这么多

假设你要给后台列表页增加筛选栏，并保持现有视觉风格。

低质量提示词：

```text
帮我把这个页面优化一下，加个筛选。
```

问题显而易见：

- 没有说明范围
- 没有说明当前阶段
- 没有说明哪些风格不能动
- 没有说明结果要求

更高质量的写法：

```text
我需要给当前后台列表页增加筛选栏。
请基于现有页面结构先分析，不要一次性重写页面。

请完成：
1. 识别当前页面结构中最适合插入筛选栏的位置
2. 说明可能需要调整的状态、表单或请求参数
3. 给出最小改动方案

约束：
- 保持现有设计风格
- 不重做整页布局
- 不修改无关组件
- 不确定的地方先列假设

输出要求：
- 用简短列表回答
- 先说结论，再说理由
```

这里真正提升质量的，不是文采，而是任务被拆清楚了。

## 迭代澄清，比一次说完更现实

很多人写提示词时有一种隐形压力：我是不是必须一开始就把所有东西说明白？

不用。

现实里更有效的方法，往往是分两轮甚至三轮逐步澄清。

例如：

- 第一轮先识别相关文件和风险点
- 第二轮补充更具体的修改目标
- 第三轮再要求给出代码或详细步骤

这种方式更符合真实开发流程。很多任务本来就是边看边清楚，而不是一开始就完全明白。

## 如何判断问题出在“上下文不够”还是“提法不准”

这是很实用的判断能力。

如果它回答很空、很泛、很像模板文，大概率是上下文不够。

如果它回答方向错了，但你其实给了不少信息，大概率是目标或约束没有写清。

如果它回答内容有用，但粒度不对，比如你要排查路径，它却给了完整重构方案，大概率是阶段说明和输出要求不够明确。

学会这样判断，后面调整提示词会快很多。

## 一套更成熟的任务描述模板

当你已经过了第一次上手阶段，可以用下面这套结构：

```text
当前目标：
[用一句话说明这次要完成什么]

当前对象：
[说明相关文件、模块、页面、接口或报错]

请你完成：
1. [分析任务一]
2. [分析任务二]
3. [输出任务三]

约束条件：
- [边界一]
- [边界二]
- [边界三]

输出要求：
- [格式要求一]
- [格式要求二]
```

这套结构的价值，在于它让你每次都按相同逻辑组织任务。久而久之，你会越来越少出现“我知道自己想做什么，但就是写不清”的情况。

## 本章总结

让 Codex 听懂你，靠的不是神秘咒语，而是任务表达能力。

你应该带走的关键点是：

- 先说目标，再补背景
- 上下文要相关，不要泛滥
- 约束条件是稳定性的核心
- 输出要求直接决定后续处理成本
- 可以通过多轮澄清逐步收紧任务

当你掌握这些之后，Codex 的输出会从“有时有用”逐渐变成“多数时候都比较稳”。

## 实战任务

找一个你最近正在做的需求，把最初脑子里的模糊描述先写出来，再把它改写成更成熟的任务描述。

至少补齐以下 4 项：

1. 目标
2. 当前对象
3. 约束条件
4. 输出要求

然后用改写后的版本去问 Codex，和原始写法做一次对比。
