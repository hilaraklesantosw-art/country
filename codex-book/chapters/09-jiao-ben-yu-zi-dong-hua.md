# 第 9 章 脚本与自动化实战，批处理、数据处理、效率工具

## 本章目标

让读者学会把 Codex 用在高回报的重复劳动场景里。

## 你将学会什么

- 如何识别值得自动化的小任务
- 如何把重复动作交给脚本
- 如何让 Codex 生成可验证的小工具
- 为什么自动化场景常常比业务开发更容易立刻见效

## 最容易被低估的用法，往往最值钱

很多人一想到 AI 编程，就只想到写业务代码。实际上，在很多真实工作流里，最先带来明显收益的往往不是业务功能，而是那些琐碎、重复、低创造性但很耗时间的动作。

比如：

- 批量整理文件名
- 批量替换文本
- 从一批数据里提取字段
- 把日志转成结构化信息
- 快速生成一个一次性小工具

这类事情看起来不显眼，但特别适合 Codex。

因为它们通常具有三个特点：

- 输入输出相对明确
- 规则相对固定
- 验证成本相对低

也就是说，非常适合做自动化。

## 如何判断一个任务值不值得自动化

你可以用一个非常简单的标准来判断：

**这个动作是不是会重复做，而且每次做起来都不难，但很烦。**

如果答案是“是”，它通常就值得考虑自动化。

例如：

- 每周都要整理一批 Markdown 文件
- 每次发布前都要做某种格式转换
- 总要从一堆文本里提取同类信息
- 经常写类似的小脚本，但每次都从零开始

这类任务最适合交给 Codex 协助，因为你很容易把规则说清楚，也很容易验证结果。

## 一个典型例子：批量整理 Markdown 文件

假设你手头有很多 Markdown 文件，命名风格混乱，有的大写，有的带空格，有的用下划线，有的缺少索引，你想做一轮统一整理。

这就是一个典型自动化场景。

很多人面对这类事情会想：“我自己也能慢慢改。”没错，但问题不在于能不能改，而在于这件事重复、机械、无聊，而且非常容易出错。

这时你可以这样让 Codex 协作：

```text
我需要处理一批 Markdown 文件，请帮我设计一个脚本。

目标：
- 统一文件命名风格
- 生成或更新索引

约束：
- 不修改文件内容
- 优先保证可回滚和可验证
- 先说明脚本逻辑，再给代码

请你完成：
1. 拆解处理步骤
2. 指出潜在风险
3. 给出脚本初稿
4. 说明如何验证执行结果
```

这个场景的关键不在于脚本有多复杂，而在于你把“目标”和“风险”说清楚了。

## 自动化场景里，验证比生成更重要

很多人让 Codex 写脚本时，会过度关注“能不能写出来”，但真正决定这类任务值不值的是：你能不能安全执行。

自动化最怕的不是写不出来，而是“一次运行，批量搞坏”。所以你要特别重视以下问题：

- 输入是什么
- 输出会变成什么
- 是否支持 dry run
- 是否能先只打印计划动作
- 是否方便回滚

当你开始主动要求这些内容时，脚本类协作的安全性会高很多。

## 自动化任务最适合的协作顺序

### 第一步：先定义输入输出

例如：

- 输入是一批文件
- 输出是统一命名后的文件和索引

这一步如果不清楚，后面的脚本很容易写偏。

### 第二步：先要求它说逻辑

不要一开始就只要代码。先让 Codex 把处理逻辑写出来。你要先确认：

- 它理解的规则是否正确
- 它有没有忽略边界情况
- 它有没有考虑回滚或验证

### 第三步：再让它出脚本

逻辑确认后，再让它生成脚本，成功率会更高。

### 第四步：最后让它补验证方案

例如：

- 先处理 3 个样例文件
- 先输出预期变更，不直接执行
- 检查命名结果和索引结果

这种顺序特别适合脚本和自动化场景。

## 除了文件批处理，还适合哪些自动化任务

### 1. 数据清洗

比如：

- 统一字段格式
- 去除脏数据
- 转换 CSV、JSON、Markdown 等格式

### 2. 日志和文本处理

比如：

- 从日志里提取关键信息
- 批量格式化一批文本
- 把零散信息整理成结构化摘要

### 3. 效率小工具

比如：

- 一次性命令行工具
- 本地辅助脚本
- 发布前检查脚本
- 小型报表整理工具

这些任务的共同点是：不需要高复杂业务判断，但非常吃重复劳动时间。

## 自动化协作里最常见的错误

### 错误一：不先说清输入输出

输入输出不明确，脚本大概率会偏。

### 错误二：不重视安全边界

比如不要求 dry run、不要求预览、不考虑回滚。这在批处理任务里非常危险。

### 错误三：只看“能跑”，不看“能不能稳定复用”

一个真正有价值的自动化工具，不只是跑通一次，而是下次还能放心继续用。

### 错误四：把自动化任务描述得太大

“帮我把所有流程自动化”这种话没有意义。你应该先抓一个最痛、最重复、最清楚的动作。

## 本章总结

自动化场景的核心收益，在于持续节省时间，而不是一次性炫技。

你应该带走这些结论：

- 最值得自动化的是重复、机械、规则清楚的任务
- 先定义输入输出，再要代码
- 安全边界和验证步骤比脚本本身更重要
- 小而实用的自动化，往往比大而空的自动化更容易产生真实收益

只要你每周能让 Codex 帮你省掉几次机械劳动，它就已经在非常实际地给你创造价值。

## 实战任务

找一个你每周都会重复做的动作，让 Codex 帮你把它脚本化。

要求你至少完成：

1. 说明输入输出
2. 先确认处理逻辑
3. 再生成脚本
4. 最后补验证步骤

只做一个小任务就够，但要真的落地。
