# 数据应用 Vibe Coding 平台 PRD

## 1. 文档概述

### 1.1 文档目的

本文档用于定义一个类似 Base44 的在线版 Vibe Coding 工具，但产品核心不面向通用网站搭建，而是面向基于数据仓库的数据应用构建。平台目标是让业务、数据、分析和运营团队通过自然语言快速生成、修改和发布数据应用。

### 1.2 产品定位

这是一个面向数据仓库场景的数据应用构建平台。用户可以通过自然语言描述需求，由平台帮助生成可运行的数据应用，包括但不限于：

- BI 大屏
- 经营分析看板
- 报表查询页
- 数据搜索页
- Data Agent 对话页
- Data Agent 执行页
- 数据运营控制台

### 1.3 核心价值

- 让非前端、非全栈用户也能快速做出数据应用
- 将“数据模型 + 数据仓库 + AI 生成界面 + Agent 执行”整合为一个平台
- 降低数据应用从需求到上线的交付成本
- 让数据团队从单次报表开发转向可复用的数据应用生产

## 2. 背景与机会

### 2.1 当前问题

企业内部的数据仓库越来越成熟，但基于数仓的数据应用交付仍存在明显瓶颈：

- BI 工具适合看图表，但不适合构建复杂的数据应用
- 自定义报表和数据页面依赖前端、后端协作，交付周期长
- Data Agent 原型容易做，但缺少统一的展示层和执行层
- 业务方知道自己要什么，但不会把需求翻译成技术实现
- 数据资产已经在仓库里，但真正被业务高频消费的应用数量有限

### 2.2 产品机会

随着大模型和生成式开发能力的发展，可以把“自然语言描述需求 -> 自动生成数据应用 -> 连接数仓 -> 迭代调整 -> 发布上线”串成一条更短的链路。对企业来说，这意味着数据消费门槛下降、数据产品化速度提升，以及更多场景能被快速验证。

### 2.3 Wren AI 开源能力调研结论

Wren AI 是当前较有代表性的开源 GenBI 产品之一，其公开站点与文档显示，它已经覆盖了从自然语言问数到图表生成、Dashboard 管理、语义层建模和嵌入式 API 的完整链路。其能力对本产品有较强参考价值。

调研得到的核心结论如下：

- Wren AI 支持通过自然语言生成 SQL、图表、报表、表格和 BI 能力，不只是单一问数工具
- Wren AI 支持自动图表生成，系统会根据问题与结果自动选择较合适的图表类型
- Wren AI 支持对图表进行编辑、重生成、恢复上一个版本，并支持导出 `PNG` 和 `SVG`
- Wren AI 支持将生成图表 pin 到 Dashboard，并支持多个 Dashboard 管理、拖拽排序和调整大小
- Wren AI 提供 API 访问能力，可将自然语言查询与图表生成嵌入到第三方产品
- Wren AI 的关键差异点不只在 UI，而在其语义层与 Wren Engine
- Wren Engine 公开文档明确强调其作用是为 AI Agent 和数据应用提供“business context”，而不是只暴露原始表结构
- Wren Engine 通过 MDL 建模数据集、关系、计算字段和业务视图，以提升 Text-to-SQL 与 Agent 数据理解的稳定性

对本产品的直接启发有四点：

1. 不能只做“自然语言生成页面”，必须做“自然语言 + 语义层 + 图表 + Dashboard + API”整链路
2. BI 生成能力不能停留在一次性出图，还要支持编辑、重生成、保存、嵌入和复用
3. Data Agent 要可靠，底层必须有业务语义层或上下文引擎，而不能只给模型裸表结构
4. 如果目标是类似 Base44 的在线构建体验，那么还应在 Wren AI 的 GenBI 基础上再往前一步，支持更完整的数据应用页面生成和执行型 Agent 场景

## 3. 产品目标

### 3.1 业务目标

- 缩短数据应用交付周期
- 提高数据仓库资产复用率
- 让业务团队更容易消费数据
- 让 Data Agent 从演示级能力变成可运行的业务工具

### 3.2 产品目标

- 提供在线、对话式的数据应用生成体验
- 提供可视化编辑与自然语言编辑结合的能力
- 提供数据仓库连接、语义层绑定和权限治理能力
- 支持 BI 大屏、查询页、Agent 页等多种数据应用形态

### 3.3 成功标准

- 用户能通过自然语言在短时间内生成一个可运行的数据应用原型
- 应用可直接接入真实数据仓库或语义层
- 平台具备发布、权限、审计、版本管理能力
- Data Agent 不仅能展示结果，还能在受控前提下执行动作

## 4. 目标用户

### 4.1 核心用户

- 数据分析师
- 数据产品经理
- BI 团队
- 运营分析团队
- 企业内部数据平台团队

### 4.2 次级用户

- 风控、供应链、财务等业务部门

- 对数据应用有较强需求

## 5. 核心使用场景

### 5.1 通过自然语言创建 BI 大屏

用户输入：

- “帮我做一个电商经营分析大屏，展示 GMV、订单数、客单价、转化率，并支持按渠道和日期筛选”

平台输出：

- 页面结构
- 图表布局
- 查询条件
- 指标绑定
- 可运行的 BI 大屏

### 5.2 创建数据查询与分析页面

用户输入：

- “做一个用户留存分析页面，可以选择时间范围、渠道和新老客，展示次留、7 留、30 留”

平台输出：

- 查询表单
- 指标图表
- 表格与导出能力
- 数据口径说明

### 5.3 创建 Data Agent 展示与执行页面

用户输入：

- “创建一个销售数据 Agent，支持问本周成交额、对比上周，并在授权范围内生成跟进任务”

平台输出：

- Agent 对话界面
- 推荐问题入口
- 数据结果展示模块
- 执行动作按钮或表单
- 审批与执行反馈机制

### 5.4 快速迭代已有数据应用

用户输入：

- “把这个经营看板改成更适合移动端，增加利润率趋势图”
- “把这个 Agent 页面加上异常预警摘要”

平台应能基于已有应用结构持续编辑，而不是每次重新生成。

## 7. 产品核心能力

### 7.1 自然语言应用生成

平台应支持用户通过自然语言描述需求，生成一类完整数据应用。

输入应能覆盖：

- 业务目标
- 页面类型
- 指标需求
- 维度与筛选条件
- 数据源
- 展示方式
- 交互行为
- 是否涉及 Agent 执行

系统应输出：

- 页面信息架构
- 数据查询结构
- 图表与组件布局
- 筛选器与交互逻辑
- 发布前预览

### 7.2 可视化编辑与对话式编辑结合

平台不应只停留在对话生成。用户需要两种编辑模式：

- 对话式编辑：用自然语言继续修改
- 可视化编辑：拖拽布局、调整组件、修改字段绑定

二者需要保持一致，避免“对话修改后页面结构失控”。

### 7.3 数据仓库连接

平台必须围绕数据仓库能力建设。

应支持连接：

- Snowflake
- BigQuery
- Redshift
- ClickHouse
- Doris
- PostgreSQL
- MySQL
- Trino/Presto

后续可扩展：

- 湖仓一体平台
- 企业语义层
- 指标平台

### 7.4 数据语义层与指标绑定

平台不应只允许直接写 SQL。更高优先级是支持语义层和指标模型绑定。

能力包括：

- 指标定义绑定
- 维度定义绑定
- 口径说明展示
- 字段权限控制
- 数据血缘信息展示

### 7.5 BI 大屏生成能力

平台需支持生成 BI 大屏类应用，包括：

- 顶部 KPI 区
- 图表区
- 趋势区
- 分渠道对比区
- 表格区
- 筛选器区

图表类型至少包括：

- 折线图
- 柱状图
- 堆叠图
- 饼图
- 漏斗图
- 表格
- KPI 卡片
- 地图

在产品形态上，需要吸收 Wren AI 已验证有效的能力：

- 自然语言出图
- 自动推荐图表类型
- 图表二次编辑
- 图表重生成
- 导出图表资产
- 图表保存与复用
- 图表 pin 到 Dashboard

但本平台不能只停留在“图表 pin 到 Dashboard”的轻量模式，而应进一步支持：

- 多页面数据应用
- 图表、表格、表单、Agent 区块混合布局
- 筛选器与组件联动
- 应用级发布与权限治理

### 7.6 Data Agent 页面能力

平台需支持生成带 Agent 交互的数据应用页面。

包括：

- 对话框
- 推荐问题
- 查询结果卡片
- 图表嵌入
- 数据解释区
- 动作执行区
- 历史对话
- 反馈与纠错

参考 Wren Engine 的能力边界，本平台的 Data Agent 底层也应建立业务语义层或上下文引擎，用于：

- 统一业务术语
- 管理指标口径
- 约束 join 路径
- 复用计算逻辑
- 提升 Text-to-SQL 稳定性
- 为 Agent 执行提供可审计的数据上下文

### 7.7 Agent 执行能力

该平台的特殊点在于 Data Agent 不只展示结果，还可执行受控动作。

执行动作示例：

- 创建分析任务
- 生成报告
- 发送通知
- 写入工单
- 提交审批
- 触发自动化流程

执行必须满足：

- 明确权限边界
- 高风险动作二次确认
- 执行日志可追踪
- 失败可回溯

### 7.8 项目与模板体系

平台应支持：

- 从空白创建
- 从模板创建
- 从已有应用克隆
- 从自然语言直接生成

模板类型建议包括：

- 经营看板模板
- 留存分析模板
- 销售分析模板
- 财务分析模板
- Agent 查询模板
- Agent 执行模板

1. 用户进入平台首页
2. 选择新建应用
3. 输入自然语言需求

1. 打开已有项目
2. 用对话或可视化方式修改
3. 预览数据结果
4. 保存草稿
5. 发布新版本

1. 用户进入 Agent 页面
2. 输入问题或选择推荐问题

4. 用户选择执行动作
5. 系统确认权限与审批
6. 执行完成后反馈结果

## 12. 技术架构要求

### 12.1 总体架构

建议采用以下分层：

- 交互层：工作台、编辑器、生成应用运行层
- 生成层：自然语言解析、页面 DSL、组件树生成
- 数据层：数据源连接、语义层、查询执行、缓存
- Agent 层：问答、工具调用、动作执行、审批
- 治理层：权限、发布、审计、日志、监控

### 12.1A 技术选型原则

本平台的技术选型应遵循以下原则：

- 优先可控，而不是完全依赖模型自由生成代码
- 优先生成中间表示层，而不是每次直接生成整站源码
- 优先围绕数据仓库、语义层和查询安全设计，而不是先做通用低代码
- 优先保证“可编辑、可发布、可回滚”，而不是只追求首次生成效果
- Agent 执行必须是受控运行，而不是开放式任意脚本执行

### 12.1B 核心判断：底层不应采用纯模型直写代码

本产品不建议采用“用户提一句话，底层让大模型直接生成 React + SQL + 后端源码并立即上线”的方案。

原因包括：

- 结果不稳定，难以保证连续迭代的一致性
- 对数据权限、查询边界和语义口径不友好
- 生成后的代码不易做可视化回改
- 应用版本管理、回滚和治理复杂
- Agent 执行动作风险难控

更合理的方案是：

- 用大模型理解用户需求
- 生成平台内部可控的页面 DSL、组件树、查询定义和 Agent 配置
- 再由平台运行时负责渲染、执行和发布

### 12.1C 推荐总体技术路线

推荐采用：

- 大模型 + 自研生成编排引擎 + 自研页面 DSL + 自研运行时

而不是：

- 纯大模型直接写整站代码
- 直接把通用 coding agent 当成线上产品底座

具体来说：

- 大模型负责理解需求、规划页面、推荐图表、生成 DSL 和查询框架
- 自研生成编排引擎负责把自然语言转成结构化应用定义
- 自研运行时负责渲染页面、执行查询、驱动 Agent 和治理发布
- 代码模型或 coding agent 只在“高级自定义逻辑”或“导出源码模式”中作为辅助能力使用

### 12.2 页面描述模型

平台需要一套统一的页面 DSL，用来描述：

- 布局
- 组件
- 数据绑定
- 交互逻辑
- 权限规则
- 发布状态

### 12.2B 前端运行时选型建议

建议前端编辑器与运行时采用：

- React
- Next.js
- TypeScript

原因：

- 组件生态成熟，适合搭建编辑器和应用运行层
- 既适合工作台，也适合生成后页面渲染
- 对服务端渲染、路由、鉴权和嵌入都更友好

页面渲染层建议分为两部分：

- 编辑器前台：负责拖拽、配置、预览
- 应用运行层：负责渲染最终页面、处理筛选和交互

图表组件建议优先支持：

- ECharts
- AntV
- AG Grid 或同类表格组件

原因：

- 适合企业数据分析场景
- 图表种类足够丰富
- 表格、看板和大屏能力成熟

### 12.2C 为什么需要页面 DSL

页面 DSL 是平台的核心资产。

建议不要让模型每次直接改 React 源码，而是先生成类似以下结构：

- 页面元数据
- 布局树
- 组件列表
- 组件属性
- 数据绑定定义
- 交互事件
- 权限配置
- 发布配置

这样可以实现：

- 对话式编辑
- 可视化编辑
- 版本 diff
- 可控发布
- 稳定导出源码

### 12.2D 推荐 DSL 结构

建议 DSL 至少分 4 层：

- Page：页面级配置
- Layout：栅格、区域、容器
- Widget：图表、表格、筛选器、文本、Agent 面板
- DataBinding：指标、维度、查询、交互联动

每个 Widget 应具备：

- 唯一 ID
- 类型
- 样式配置
- 数据源引用
- 查询定义
- 权限要求
- 事件定义

### 12.2E 是否需要源码导出

建议支持“导出源码”，但不把它作为平台主运行方式。

推荐方式：

- 平台默认运行在托管 DSL runtime 上
- 企业高级版或开发者模式支持导出 React/Next.js 源码
- 导出源码时，可调用 coding model 辅助补齐页面代码与测试

这样既能保证平台可控，也能满足企业二次开发诉求

### 12.2A 语义层与上下文引擎要求

结合 Wren AI 的公开架构经验，本平台需要单独建设语义层或上下文引擎，而不应让大模型直接面对原始数仓 Schema。

该层至少要支持：

- 业务实体定义
- 指标定义
- 维度定义
- 关系与 join path 定义
- 计算字段定义
- 业务视图抽象
- 版本化管理
- 对 Agent 与页面生成统一暴露上下文

这一层的作用包括：

- 提高自然语言生成 SQL 的可靠性
- 提高图表生成与指标绑定的一致性
- 降低模型对原始表结构的误解
- 让 Data Agent 的查询与执行建立在统一业务语义之上

### 12.2B1 语义层技术路线建议

语义层建议优先采用“独立服务 + 标准定义文件”的方式建设。

建议能力形态：

- 语义模型管理服务
- 指标与维度注册服务
- 查询编译服务
- 口径说明服务

模型定义可采用：

- YAML/JSON 定义文件
- 数据建模 DSL
- 后台管理配置界面

第一版不建议直接支持任意裸 SQL 暴露给终端用户，而应：

- 管理员或分析师定义可复用语义对象
- 页面和 Agent 基于语义对象生成查询
- 对必要的 SQL 自定义提供白名单式扩展

### 12.2B2 查询执行层选型建议

查询执行层建议单独做服务，不要散落在前端页面逻辑中。

建议能力包括：

- 连接池管理
- 查询超时
- 查询缓存
- 查询重试
- 查询审计
- 行列权限注入
- 返回结果标准化

如果数据源复杂，可以考虑：

- 统一查询网关
- 支持同步查询与异步查询
- 大结果集分页或导出

### 12.2B3 缓存与性能建议

针对 BI 与分析页面，需要单独设计缓存层。

建议分三类缓存：

- 语义元数据缓存
- 查询结果缓存
- 图表配置缓存

对于高频大屏场景，可支持：

- 定时刷新
- 预热查询
- 快照缓存

### 12.3 生成引擎选型建议

生成引擎建议采用“多阶段编排”模式，而不是单轮 prompt 直接出最终页面。

建议拆成以下步骤：

1. 意图解析
2. 页面规划
3. 语义对象匹配
4. 查询结构生成
5. 图表与组件选择
6. 页面 DSL 组装
7. 预览校验
8. 用户确认与发布

每一步都应有中间结果，便于：

- 回滚
- 局部编辑
- 调试
- 审计

### 12.3A 大模型选型建议

大模型层建议至少分成两类模型：

- 推理/规划模型
- 代码/配置生成模型

推理模型负责：

- 需求理解
- 页面结构规划
- 图表推荐
- 语义对象选择
- Agent 行为规划

代码或配置生成模型负责：

- 页面 DSL 输出
- 查询模板生成
- 源码导出辅助
- 测试代码和连接器样板生成

### 12.3B 是否需要 OpenCode 或类似 coding agent

不建议把 OpenCode、Codex CLI、或类似 coding agent 直接作为线上产品底座。

更合理的定位是：

- 作为平台内部研发提效工具
- 作为“源码导出模式”的辅助生成器
- 作为复杂自定义组件开发的工程助手

原因：

- 这类工具擅长改代码仓库，不擅长做在线多用户产品运行时
- 平台真正核心是 DSL、语义层、运行时和治理体系
- 如果把线上核心链路交给通用 coding agent，稳定性和可控性都不足

### 12.3C 模型调用架构建议

建议模型层采用统一模型网关，而不是在多个服务里直接散连模型厂商。

统一模型网关应支持：

- 多模型路由
- Prompt 模板管理
- 成本监控
- 调用日志
- 限流
- 回退策略
- 模型版本管理

这样便于后续切换：

- OpenAI
- Azure OpenAI
- Anthropic
- Gemini
- 开源自部署模型

### 12.4 后端服务选型建议

后端建议采用服务化设计，推荐组合为：

- API 层：TypeScript Node.js 或 Python FastAPI
- 任务编排层：Python 更适合模型编排与数据处理
- 查询网关层：独立服务
- Agent 执行层：独立服务

如果团队偏前端和全栈，可采用：

- Next.js + Node.js 为主

如果团队偏数据与 AI 工程，可采用：

- 前台 Next.js
- 后台 FastAPI / Python

从当前产品形态看，更推荐：

- 前端与 BFF：Next.js / TypeScript
- 生成编排与 Agent：Python / FastAPI
- 查询与权限网关：独立服务

### 12.4A 存储选型建议

平台自身除外部数据仓库外，还需要自己的元数据存储。

建议存储拆分为：

- PostgreSQL：项目、页面、版本、权限、发布记录
- Redis：缓存、会话、短期状态
- Object Storage：导出文件、静态资源、应用快照
- 日志系统：审计、执行日志、模型调用日志

### 12.4B Agent 执行层选型建议

Agent 执行层不能和普通问答混在一起。

建议拆分为：

- Query Agent：负责问数与解释
- Action Agent：负责执行动作
- Approval Service：负责审批流
- Execution Log Service：负责记录执行链路

对于执行动作，建议采用：

- 工具注册表
- 白名单动作
- 参数校验
- 角色权限控制
- 幂等控制

第一版严格限制动作为：

- 生成报告
- 发送通知
- 创建任务
- 提交审批

不建议一开始支持高风险写库或跨系统事务动作

### 12.3 Agent 描述模型

平台需要一套 Agent 配置模型，用来描述：

- 输入意图
- 数据查询能力
- 展示组件
- 可执行动作
- 权限边界
- 审批流程

### 12.3A1 Agent 运行协议建议

Agent 配置建议至少包含：

- 意图分类
- 可访问语义对象
- 可调用工具
- 响应展示组件
- 可执行动作
- 风险等级
- 确认策略
- 审批策略

这样可以把“展示型 Agent”和“执行型 Agent”明确区分开，降低风险

### 12.4 API 与嵌入能力

参考 Wren AI 对外提供 SQL 生成与图表生成 API 的能力，本平台也应支持 API 与嵌入式交付。

建议第一阶段 API 覆盖：

- 自然语言转查询
- 查询执行
- 图表规范生成
- 应用运行态嵌入
- Agent 问答调用
- Agent 执行动作调用

如果本平台未来希望成为企业数据应用基础设施，而不仅是一个内部页面生成器，那么 API 与嵌入能力必须在架构上前置考虑。

### 12.4A1 嵌入式交付建议

嵌入式交付建议支持：

- iframe 嵌入
- SDK 嵌入
- Token 鉴权嵌入
- 单点登录嵌入

如果未来支持企业门户、CRM、ERP、OA 等场景集成，SDK 能力会比单纯 iframe 更有价值。

## 14. MVP 建议

第一版 MVP 建议聚焦以下闭环：

- 自然语言生成 BI 页面
- 支持至少 2 到 3 种主流数据源
- 提供模板化大屏与分析页
- 提供基础 Agent 问答页面
- 支持受限执行一个简单动作
- 支持草稿、预览、发布

先不做：

- 复杂多智能体协作
- 复杂跨系统事务执行
- 高自由度视觉设计器
- 海量第三方模板市场

## 15. 里程碑建议

### Phase 1：原型验证

- 完成自然语言生成页面原型
- 接通基础数据源
- 支持简单 BI 大屏生成

### Phase 2：数据应用闭环

- 增加对话式编辑
- 增加发布与权限
- 增加 Agent 查询与结果展示

### Phase 3：执行与治理增强

- 增加 Agent 执行动作
- 增加审批、审计和回滚
- 增加模板与团队协作能力

## 16. 风险与难点

- 自然语言生成的数据页面如果不稳定，会影响用户信任
- 数据仓库与语义层接入复杂度高
- Agent 执行能力如果权限设计不清晰，风险很大
- 如果平台只做到“能生成”，但不好改、不好发布，长期价值会受限
- 业务用户与数据团队对口径一致性的要求很高
- 如果没有类似 Wren Engine 这样的上下文层，模型对 Schema、指标和关系的理解会不稳定，导致 BI 和 Agent 结果失真
- 如果只提供单轮出图而不支持 Dashboard 管理、图表复用和嵌入能力，产品价值会停留在 Demo 层

### 17.1 Wren AI 参考结论

- GenBI：自然语言生成 SQL、图表、报表、表格与 BI
- Dashboard：支持将图表 pin 到 Dashboard，并管理多个 Dashboard

- Semantic Engine：通过 Wren Engine 和 MDL 提供业务语义与上下文层

本产品与 Wren AI 的关系不应是简单复制，而应是：

- 在其 GenBI 和语义层经验之上
- 进一步扩展到完整数据应用生成
- 以及带执行能力的 Data Agent 页面
