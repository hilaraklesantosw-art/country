## 2. 一句话定义

Next Data 是一个面向企业经营分析与业务决策的下一代数据应用平台。平台基于既有指标、数据集与语义模型，理解用户一句话业务诉求，自动生成可编辑、可追问、可分享、可治理的高品质 BI 看板与数据应用页面，显著降低数据分析和看板交付成本。

## 3. 背景与问题定义

### 3.1 业务背景

企业已经沉淀了大量指标、宽表、数据集、标签、主题域和分析语义，但数据消费仍高度依赖数据产品经理、BI 开发、分析师和可视化工程师协同完成。典型问题如下：

1. 从业务需求到成品看板交付周期长，通常需要 3 天到 3 周。
2. 业务方难以准确表达图表结构、钻取关系、过滤条件与页面布局。
3. 数据资产虽多，但缺乏统一的语义消费入口，导致“有数不会用、有表找不到”。
4. 传统 BI 偏“拖拽式搭建”，对非专业用户门槛高，对专业用户又不够高效。
5. 看板交付后静态化严重，迭代依然依赖人力，无法自然支持“继续追问”和“按场景快速改版”。
6. 企业级分享能力不足，缺少权限裁剪、外发控制、版本管理、审计和回溯。

### 3.2 核心问题

本项目要解决的不是“做一个更会画图的 BI”，而是构建一个“从业务意图到数据应用”的自动化生产系统。其本质是将企业数据资产、语义层、可视化能力和大模型推理能力整合为统一产品能力，做到：

1. 用户能用自然语言表达业务目标，而非学习 BI 工具。
2. 系统能自动将业务目标转译成指标选择、分析路径、页面结构与交互逻辑。
3. 生成结果不是一次性图片，而是可编辑、可运营、可治理的正式数据应用资产。

## 4. 产品目标

### 4.1 北极星目标

让企业用户在 5 分钟内，从一句话业务需求生成一个可用、可编辑、可分享的高质量 BI 看板。

### 4.2 业务目标

1. 将看板交付效率提升 10 倍以上。
2. 将普通业务用户的数据应用生成门槛降低到“会描述问题”即可。
3. 提升企业数据资产复用率与语义层调用率。
4. 建立 AI 驱动的数据应用生产范式，为后续智能分析、经营驾驶舱、智能预警、经营复盘提供底座。

### 4.3 用户目标

1. 更快拿到可直接用于汇报和决策的科技感看板。
2. 看板内容具备业务解释性，知道“为什么是这些指标和图”。
3. 支持继续改、继续问、继续分享，而非一次性生成。

### 4.4 非目标

1. 不在 V1 解决任意复杂 SQL 手写能力。
2. 不在 V1 覆盖企业全部可视化组件生态。
3. 不在 V1 替代底层数仓、指标平台、权限平台和元数据平台。
4. 不在 V1 支持对外公开互联网匿名分享。

## 5. 目标用户与场景

### 5.1 目标用户

1. 业务负责人：关注经营结果、异常定位、团队汇报。
2. 运营负责人：关注活动效果、转化漏斗、用户增长与留存。
3. 分析师：关注高效搭建、复用模板、快速迭代和深度分析。
4. 数据产品经理：关注指标治理、语义统一、交付提效。
5. 管理层：关注统一口径、简洁结论、移动端查看与分享。

### 5.2 核心场景

1. 一句话生成经营驾驶舱。
2. 针对某业务主题生成专题分析页。
3. 基于已有看板继续重构为更适合汇报的版本。
4. 按组织、角色、客户、渠道生成差异化视图。
5. 将数据看板作为链接、快照、嵌入卡片进行协作传播。

### 5.3 示例需求

“帮我生成一个科技感强、适合 CEO 周会汇报的电商经营看板，重点看 GMV、订单转化率、新客占比、投放 ROI 和区域表现，并突出异常波动。”

## 6. 产品价值主张

### 6.1 对业务侧

从“提需求等交付”升级为“直接生成并迭代”。

### 6.2 对数据团队

从“重复做页面”升级为“沉淀语义、治理资产、训练模板和规则”。

### 6.3 对管理层

从“拿到报表”升级为“拿到可追问、可分享、可持续迭代的数据应用”。

## 7. 产品原则

1. 业务意图优先：系统首先理解业务目标，其次才是图表。
2. 语义驱动优先：所有自动生成必须基于企业统一指标和语义层，不允许模型自由编造数据定义。
3. 可编辑优先：生成结果必须进入正式编辑器，而不是仅给出静态成品。
4. 高质量优先：默认输出需达到“可用于高层汇报”的视觉和内容标准。
5. 可解释优先：系统需说明指标选取逻辑、口径来源和生成理由。
6. 安全合规优先：权限、审计、脱敏、分享边界必须内建，而不是后补。

## 8. 整体方案概述

平台整体由六层能力组成：

1. 数据资产层：指标平台、数据集、标签、维表、元数据、权限系统。
2. 语义理解层：统一业务语义、指标口径、维度关系、同义词和领域知识。
3. AI 规划层：基于用户需求完成意图识别、需求补全、分析规划、页面规划和组件编排。
4. 数据查询层：将规划结果映射为合规查询计划，调用指标服务或查询引擎获取数据。
5. 应用生成层：自动生成看板布局、图表、说明文案、筛选器、交互动作和主题样式。
6. 协同分享层：支持编辑、版本、审批、权限分享、嵌入、导出和审计。

## 9. 核心用户旅程

### 9.1 主链路

1. 用户输入一句话需求。
2. 系统识别业务主题、对象、时间范围、关键指标、目标受众、页面风格和潜在缺失项。
3. 系统自动补全需求，并以可确认的“生成计划”形式展示。
4. 用户确认后，系统基于语义层选择指标、维度和数据集，生成分析计划。
5. 系统输出可编辑看板，包括布局、图表、卡片说明、筛选器和标题。
6. 用户可以继续通过自然语言修改，也可以在编辑器中手工微调。
7. 用户保存、发布、分享、嵌入或导出。

### 9.2 关键体验要求

1. 首次生成过程需要“快而稳”，默认 60 秒内完成。
2. 用户永远知道系统用了哪些指标、哪些口径、哪些数据源。
3. 如果需求含糊，系统要主动补全而不是直接生成错误结果。
4. 如果没有权限或没有数据，系统必须给出明确原因和替代方案。

## 10. 功能范围

### 10.1 模块一：自然语言需求输入

#### 10.1.1 功能描述

支持用户通过一句话、多轮对话或粘贴业务背景描述生成数据应用。

#### 10.1.2 输入能力

1. 自然语言文本输入。
2. 预置需求模板快捷输入。
3. 上传已有看板链接或截图作为参考风格。
4. 指定受众，例如“CEO 汇报”“区域经理复盘”“投放运营日报”。

#### 10.1.3 系统解析要素

1. 行业/业务域。
2. 分析主题。
3. 指标诉求。
4. 时间粒度和时间范围。
5. 对比逻辑。
6. 异常关注点。
7. 风格偏好。
8. 输出形态。

### 10.2 模块二：AI 需求补全与生成计划

#### 10.2.1 功能描述

系统将模糊需求结构化，形成可执行的“生成计划”。

#### 10.2.2 生成计划内容

1. 业务目标定义。
2. 目标受众。
3. 推荐指标清单。
4. 推荐维度与筛选项。
5. 页面数量与布局建议。
6. 图表类型建议。
7. 风格主题建议。
8. 数据口径来源。
9. 风险提示与待确认项。

#### 10.2.3 交互规则

1. 缺少关键条件时，系统发起最少轮次追问。
2. 对常见场景默认补足时间范围、同比环比和核心维度。
3. 用户可一键确认、局部修改或重新生成。

### 10.3 模块三：语义映射与数据规划

#### 10.3.1 功能描述

将业务意图准确映射到企业现有指标、维度、数据集和权限体系。

#### 10.3.2 能力要求

1. 指标同义词识别，例如“成交额”映射“GMV”。
2. 口径冲突识别，例如“支付 GMV”与“下单 GMV”。
3. 时间口径识别，例如自然周、财务月、活动期。
4. 维度关联推导，例如区域、渠道、品类、客户等级。
5. 数据源优先级策略，优先使用认证数据资产。
6. 权限裁剪，用户只能使用有权访问的数据资产。

#### 10.3.3 输出结果

1. 选中指标列表。
2. 选中维度列表。
3. 数据集引用关系。
4. 查询计划。
5. 不确定项列表。

### 10.4 模块四：自动生成科技感 BI 看板

#### 10.4.1 功能描述

系统自动生成符合企业审美标准和汇报场景要求的数据看板。

#### 10.4.2 页面组成

1. 标题区：主题、时间范围、版本信息。
2. KPI 区：核心指标卡、同比环比、目标达成。
3. 分析区：趋势、构成、对比、漏斗、地图、明细。
4. 结论区：AI 自动总结洞察与风险提示。
5. 控件区：筛选器、时间切换、下钻入口。

#### 10.4.3 视觉要求

1. 默认输出须具备科技感、高级感、汇报感。
2. 支持品牌主题、暗色主题、大屏主题、汇报主题。
3. 支持自动使用一致的栅格、间距、字体层级和高亮色。
4. 支持避免“组件堆砌感”，优先输出结构清晰、留白合理的页面。

#### 10.4.4 内容要求

1. 每个图表必须有清晰标题和口径说明。
2. 重要图表优先突出异常和变化。
3. 默认包含 1 段 AI 结论摘要，但必须标记为“AI 生成”。
4. 当数据不足时，不允许生成误导性图表。

### 10.5 模块五：可编辑看板编辑器

#### 10.5.1 功能描述

用户可在 AI 生成结果基础上继续编辑，编辑方式包括自然语言编辑和可视化编辑。

#### 10.5.2 编辑能力

1. 修改标题、说明、图表类型和布局。
2. 增删图表组件。
3. 调整主题、配色、间距和卡片样式。
4. 新增筛选器和联动规则。
5. 调整指标、维度和排序规则。
6. 锁定组件，防止后续 AI 改写。

#### 10.5.3 对话式编辑示例

1. “把区域表现改成地图和排行榜双视图。”
2. “首页只保留 CEO 最关心的 6 个指标。”
3. “把整体风格改成更像未来感驾驶舱。”
4. “增加一个新客首购转化漏斗。”

### 10.6 模块六：洞察与解释

#### 10.6.1 功能描述

在生成结果中补充“为什么这样设计”和“数据说明”，提升可信度与可读性。

#### 10.6.2 输出内容

1. 指标来源与口径。
2. 图表选择原因。
3. 异常波动解释建议。
4. 推荐后续追问问题。

### 10.7 模块七：分享与发布

#### 10.7.1 功能描述

支持将数据应用高效、安全地分发给组织内部相关人员。

#### 10.7.2 分享方式

1. 链接分享。
2. 组织内协同分享。
3. 权限组分享。
4. 嵌入到门户、OA、CRM、飞书、钉钉等内部系统。
5. 导出图片/PDF 作为汇报附件。

#### 10.7.3 分享控制

1. 查看权限控制。
2. 行列级数据权限继承。
3. 链接有效期。
4. 是否允许下载。
5. 是否允许二次分享。
6. 操作审计与访问日志。

### 10.8 模块八：模板与资产沉淀

#### 10.8.1 功能描述

将高质量看板沉淀为可复用模板，提高后续生成质量和行业适配速度。

#### 10.8.2 模板类型

1. 行业模板。
2. 主题模板。
3. 汇报模板。
4. 角色模板。
5. 企业品牌模板。

## 11. 关键能力设计

### 11.1 AI 生成链路

1. 需求理解：识别意图、对象、场景、受众。
2. 需求补全：基于领域知识填补缺失项。
3. 语义对齐：映射指标、维度、数据集、权限。
4. 分析规划：决定页面结构、分析顺序、图表组合。
5. 数据取数：调用指标服务或查询引擎。
6. 页面生成：生成布局、组件、文案、样式。
7. 结果解释：输出依据、洞察、风险。
8. 可编辑化：进入正式编辑模型并保存为资产。

### 11.2 生成质量控制

1. 规则护栏：业务口径、禁用指标组合、视觉规范、数据最小样本量。
2. 资产白名单：优先使用认证指标和认证数据集。
3. 结果校验：校验空数据、异常值、重复图表、冲突口径。
4. 人机协同：高风险场景要求二次确认。
5. 回放与评估：记录 prompt、规划结果、查询结果和用户编辑行为，用于持续优化。

### 11.3 设计系统要求

1. 需建立专用 BI 设计系统，而非通用后台组件拼装。
2. 组件需支持暗背景、高亮数据、动画节奏和大屏适配。
3. 主题系统需支持品牌色注入、企业 Logo、字体和封面样式。

## 12. 角色权限模型

### 12.1 角色定义

1. 超级管理员：管理全局配置、模型策略、模板和权限。
2. 数据管理员：管理指标、语义、数据集映射和资产认证。
3. 普通编辑者：生成、编辑、发布自己的看板。
4. 查看者：查看已授权看板。
5. 模板管理员：维护行业模板和品牌模板。

### 12.2 权限原则

1. 生成权限不等于数据访问权限。
2. 分享权限不等于下载权限。
3. 模型只能访问用户可见的数据元信息和数据内容。

## 13. 非功能需求

### 13.1 性能

1. 首次生成 P90 小于 60 秒。
2. 二次编辑生成 P90 小于 20 秒。
3. 页面打开首屏 P90 小于 3 秒。
4. 分享链接访问成功率大于 99.9%。

### 13.2 可用性

1. 核心服务 SLA 不低于 99.9%。
2. 关键任务失败后支持重试与降级。
3. 模型不可用时，可退化为模板生成模式。

### 13.3 安全与合规

1. 全链路审计。
2. 敏感字段脱敏。
3. 行列级权限继承。
4. 分享访问控制。
5. Prompt 与结果可追溯。
6. 支持企业私有化或专有云部署预留。

### 13.4 可观测性

1. 记录生成耗时、失败原因、命中模板、命中语义资产情况。
2. 记录用户编辑行为和发布行为。
3. 建立质量看板，跟踪生成成功率、采用率、二次编辑率和分享率。

## 14. 数据与算法要求

### 14.1 数据前提

1. 企业已具备基础指标平台或语义层。
2. 数据集需有主题域、负责人、更新时间和质量评级。
3. 关键指标需有认证口径。

### 14.2 算法能力

1. 领域意图识别。
2. 指标与维度召回排序。
3. 页面规划和组件编排。
4. 结果解释生成。
5. 质量评估和用户反馈学习。

### 14.3 模型策略

1. 大模型负责意图理解、规划与文案解释。
2. 规则引擎负责口径、安全、权限和硬约束。
3. 推荐模型负责模板选择、图表排序和布局优化。

## 15. 关键业务规则

1. 所有自动生成内容必须可追溯到具体数据资产。
2. 当同名指标存在多个口径时，必须显式提示用户选择。
3. 当用户无权限访问核心数据时，不展示真实值，只展示权限提示。
4. AI 结论不允许伪造因果关系，必须以“可能原因”表达。
5. 发布前必须通过查询合法性和权限校验。

## 16. V1 范围定义

### 16.1 V1 必做

1. 一句话生成单页 BI 看板。
2. 支持企业已有指标和数据集接入。
3. 支持语义映射、查询计划、自动图表推荐。
4. 支持自然语言二次修改。
5. 支持可视化编辑器基础能力。
6. 支持组织内分享、链接分享、图片/PDF 导出。
7. 支持基础模板库、品牌主题和审计日志。

### 16.2 V1 可选

1. 多页专题分析册。
2. 移动端专属视图。
3. 嵌入第三方系统的低代码配置。

### 16.3 V1 不做

1. 对外匿名公开分享。
2. 任意 SQL Copilot。
3. 复杂多主体审批流。
4. 实时流式图表自动生成。

## 17. 里程碑规划

### 17.1 M0：方案准备期（2-4 周）

1. 明确试点业务域。
2. 盘点指标、数据集和语义资产现状。
3. 明确设计系统和技术架构边界。

### 17.2 M1：最小可用版本 MVP（6-8 周）

1. 支持单场景一句话生成。
2. 支持 10-20 个认证指标接入。
3. 支持 3 套主题模板。
4. 支持保存、编辑、分享。

### 17.3 M2：试点业务上线（8-12 周）

1. 覆盖 2-3 个核心业务域。
2. 引入模板沉淀和反馈学习。
3. 打通权限与审计体系。

### 17.4 M3：平台化阶段（12 周以上）

1. 支持多业务域扩展。
2. 支持模板市场和组织级资产治理。
3. 支持嵌入式数据应用交付。

## 18. 成功指标

### 18.1 业务指标

1. 看板平均交付周期下降 80%。
2. 自助生成占新增看板数比例大于 50%。
3. 看板分享率大于 40%。
4. 试点业务负责人满意度大于 85%。

### 18.2 产品指标

1. 一次生成成功率大于 75%。
2. 二次编辑后可发布率大于 60%。
3. 生成看板 7 日活跃率大于 30%。
4. 模板复用率大于 35%。

### 18.3 技术指标

1. 认证指标命中率大于 90%。
2. 错口径投诉率低于 1%。
3. 权限事故为 0。

## 19. 风险与对策

### 19.1 语义资产不足

风险：模型能理解需求，但企业指标和语义层不完善，导致生成结果不稳定。

对策：先限定试点业务域，仅开放认证资产；同步推进指标治理。

### 19.2 生成结果“好看但不好用”

风险：视觉强但分析结构弱，无法支撑业务决策。

对策：建立页面结构规则和行业模板，由数据产品经理参与共建。

### 19.3 幻觉与错误解释

风险：模型输出错误洞察或夸大因果关系。

对策：结论与解释强制引用数据事实；高风险解释以建议语气呈现。

### 19.4 权限与分享风险

风险：分享链路导致越权查看。

对策：所有分享统一走权限网关和审计中心，禁止绕过原始数据权限。

### 19.5 组织推动阻力

风险：分析师担心被替代，业务方对 AI 结果不信任。

对策：强调“AI 提效而非替代”，并通过可解释、可编辑、可治理机制建立信任。

## 20. 验收标准

### 20.1 产品验收

1. 用户输入一句话需求后，可在 60 秒内生成完整单页看板。
2. 看板至少包含 KPI、趋势、构成/对比、筛选器、AI 结论模块。
3. 用户可通过自然语言完成至少 5 类修改操作。
4. 用户可保存、发布并分享给组织内其他成员查看。

### 20.2 数据验收

1. 看板中所有指标均可查看来源和口径说明。
2. 权限不足的数据不可被越权生成或分享。
3. 同一指标在不同页面口径一致。

### 20.3 体验验收

1. 默认视觉质量达到管理汇报可用水平。
2. 页面布局一致、信息密度合理、重点突出。
3. 编辑器操作可在 10 分钟内被新用户理解并完成首次发布。

## 21. 开放问题

1. 企业当前是否已有成熟的指标平台与语义层，还是需要本项目补建轻量语义层。
2. 首批试点业务域优先选择电商经营、销售管理，还是投放分析。
3. 视觉设计系统是统一集团品牌，还是按 BU 可配置。
4. 大模型部署策略采用公有云 API、专有云还是完全私有化。

## 22. 建议的首期落地策略

建议采用“一个高价值业务域 + 一组认证指标 + 一类高层汇报场景”切入，先打穿从需求输入、语义映射、自动生成、编辑、分享的完整闭环，而不是一开始追求全行业、全场景、全组件覆盖。最优先建议试点“经营驾驶舱”或“电商周会汇报”场景，因为该场景价值感知强、用户层级高、模板复用度高，最容易形成组织影响力。

## 23. 结论

Next Data 不是一个传统 BI 工具的增强版，而是企业数据消费入口的重构。它的成败不取决于模型是否足够“聪明”，而取决于是否把企业现有的数据资产、语义治理、权限体系、设计系统和 AI 能力真正整合为一个可持续的平台级产品。一旦打通，该平台将成为企业“从业务问题到数据应用”的标准生产线，具备极强的组织杠杆价值和平台战略价值。

## 24. 补充资料：开源项目 Wren AI 调研

### 24.1 调研目的

调研 Wren AI 的目标，不是简单判断其是否可替代 Next Data，而是识别其在 GenBI、语义层、Text-to-SQL、图表生成与开源产品策略上的成熟做法，为 Next Data 的产品设计和技术路径提供参考。

### 24.2 项目概况

根据 Wren AI 官方 GitHub 仓库与官方文档，Wren AI 是一个开源的 GenBI（Generative BI）产品，核心定位是让用户通过自然语言查询数据库，并自动生成 SQL、图表、报表和 AI 洞察。其官方仓库名为 `Canner/WrenAI`，官方描述强调以下能力：

1. 用自然语言查询数据库。
2. 自动生成较准确的 Text-to-SQL。
3. 自动生成图表与 AI 洞察。
4. 依赖语义层提升可解释性与准确率。

从公开信息看，Wren AI 的开源活跃度较高。截至本次调研日期 2026-03-17，GitHub 展示其为 Public 仓库，采用 `AGPL-3.0` 许可证，星标约为 `13.4k`，最近可见发布版本为 `0.29.1`，发布日期为 `2025-11-28`。以上信息来自 GitHub 仓库公开页，属于调研时点快照。

### 24.3 官方能力范围

结合官方 GitHub、官网 OSS 页与官方文档，Wren AI 已公开提供以下产品能力：

1. Text-to-SQL：用户通过自然语言提问，由系统生成 SQL 并返回结果。
2. Text-to-Chart：基于查询结果自动生成图表。
3. GenBI Insights：自动生成摘要、报表与业务洞察。
4. Semantic Layer：通过 MDL 维护业务语义、指标、关系与计算逻辑。
5. 多数据源接入：公开列出的数据源包括 BigQuery、Snowflake、PostgreSQL、MySQL、ClickHouse、Databricks、DuckDB、Redshift、Trino、SQL Server、Oracle 等。
6. 多模型支持：官方说明支持 OpenAI、Azure OpenAI、Anthropic、Gemini、Bedrock、Databricks、Ollama、Groq、DeepSeek 等模型接入。

基于官方能力描述可以判断，Wren AI 的产品重点是“让用户与数据对话并快速产出 SQL、图表和分析结果”，其核心优势在于把 LLM 与语义层、查询引擎和交互式界面组合成一套完整 GenBI 方案。

### 24.4 技术架构与关键设计

根据官方文档《How Wren AI works?》，Wren AI 由三类核心服务组成：

1. `Wren UI`：负责用户交互、数据关系配置与接入界面。
2. `Wren AI Service`：负责上下文检索和 LLM 推理，生成更准确的 SQL。
3. `Wren Engine`：负责语义引擎、元数据管理、数据源连接和 SQL 方言转换。

该架构说明 Wren AI 不是一个单纯的 Prompt 套壳产品，而是一个围绕语义层构建的工程化系统。其技术上最值得重视的是 `Wren Engine` 与 `MDL`。

### 24.5 MDL 与语义层启发

Wren AI 官方长期强调 `MDL`（Modeling Definition Language）是其准确率与可治理性的基础。根据官方文档和官方博客：

1. MDL 用来定义实体、属性、指标、维度和表关系。
2. 业务逻辑、计算口径和关联关系通过 MDL 固化，而不是临时依赖模型猜测。
3. 语义层帮助降低 Text-to-SQL 幻觉，减少因底表结构复杂、命名不直观而导致的错误。
4. 当底层表结构变化时，只需维护语义层，而非重做全部自然语言能力。

这对 Next Data 的直接启发非常明确：如果没有企业统一语义层，再强的大模型也只能生成“看起来合理”的页面，无法生成“可用于经营决策”的正式看板。Next Data 想要达到企业级可信生成，必须优先建设或接入指标语义层，而不是先做炫酷的页面生成。

### 24.6 Wren SQL 与多数据源抽象

Wren AI 官方文档还公开了 `Wren SQL` 概念。其核心思路是对不同数据库方言做一层统一抽象，再由引擎转译为目标数据库 SQL。这意味着：

1. 上层自然语言和语义模型不必直接面对复杂的方言差异。
2. 平台更容易扩展多种数据源。
3. 统一查询语义后，更利于调试、审计和跨环境迁移。

对 Next Data 的参考意义在于：如果未来要做跨引擎、跨主题域的数据应用平台，查询计划层最好存在统一中间表示，而不是把模型直接耦合到各数据库方言。是否采用类似 Wren SQL 的形式可以再评估，但“统一中间语义查询层”值得作为架构原则保留。

### 24.7 与 Next Data 的相似点

Wren AI 与 Next Data 在方向上存在明显重合，主要体现在：

1. 都试图把自然语言变成可执行的数据分析结果。
2. 都依赖语义层来提升准确率和一致性。
3. 都不仅输出 SQL，还强调图表和分析结果的可读性。
4. 都在试图降低非技术用户使用数据的门槛。

这说明 Next Data 的方向是成立的，且已有市场验证信号。尤其是“语义层 + LLM + BI 交付”的路径，不是孤立设想，而是已被开源生态证明具有真实需求。

### 24.8 与 Next Data 的关键差异

尽管方向相近，但从目前公开资料看，Wren AI 与 Next Data 的产品重心并不完全相同。

#### 24.8.1 Wren AI 更偏“对话式数据分析平台”

Wren AI 当前公开重点更像：

1. 聊天式查询。
2. SQL 生成与解释。
3. 图表、报表和摘要生成。
4. 语义层驱动的数据探索。

其核心价值偏“分析助手”与“数据问答入口”。

#### 24.8.2 Next Data 更偏“数据应用生产平台”

Next Data 的目标不只是回答问题，而是要基于一句话业务诉求，自动生成一个可编辑、科技感、高层汇报可用、可分享的数据应用或 BI 看板，并把结果沉淀为正式资产。这里包含几个 Wren AI 当前公开描述中不够突出的部分：

1. 面向管理层汇报场景的高品质页面生成。
2. 面向业务团队协作的看板编辑器与发布流程。
3. 面向组织传播的分享、嵌入、权限裁剪和版本治理。
4. 面向品牌和设计系统的一致化视觉生成。
5. 面向经营驾驶舱的页面级信息架构与叙事编排。

因此，Wren AI 对 Next Data 的价值更像“底层方向参考”和“技术能力对标对象”，而不是完整产品形态的直接替代品。

### 24.9 对 Next Data 的产品启发

基于调研，建议将以下内容纳入 Next Data 的产品和技术建设优先级：

1. 把“企业语义层 / 指标层”前置为平台第一性能力，而非后置增强项。
2. 建立统一查询计划中间层，降低模型与具体数据源方言的直接耦合。
3. 对 AI 输出建立规则护栏和可追溯链路，避免只追求生成速度。
4. 将“对话式分析”与“正式看板生产”拆成两个层次能力，前者负责探索，后者负责交付。
5. 建立模板体系与设计系统，使生成结果不止“能看”，而是“能汇报、能传播、能复用”。
6. 从试点业务域切入，优先接入少量认证指标和高频高价值场景，避免一开始做全量通用平台。

### 24.10 对 Next Data 的竞争判断

基于官方资料做出的判断如下：

1. Wren AI 已证明“语义层驱动的 GenBI”具备较强产品吸引力和开源传播力。
2. 如果 Next Data 只做“自然语言问数 + 自动图表”，则与 Wren AI 的差异化不够明显。
3. Next Data 必须把差异化重心放在“企业级数据应用生成”和“高质量看板交付链路”上。
4. Next Data 的胜负手不应只是模型能力，而应是“语义可信 + 页面高级感 + 可编辑交付 + 权限治理 + 分享传播”的组合优势。

### 24.11 补充结论

Wren AI 的成熟之处在于，它把 LLM 与语义层、查询引擎、图表和报表输出做成了完整闭环，这证明 Next Data 的方向具备现实基础。但 Next Data 如果想形成真正的平台级竞争力，不能停留在“问答式 BI”层面，而应向上走到“数据应用自动生产”层，重点建设页面级编排能力、设计系统、组织级分享治理和经营汇报场景的强适配能力。
