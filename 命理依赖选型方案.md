# 命理依赖选型方案：`kerykeion`、`cnlunar` 与竞品分析

更新时间：2026-03-21

## 1. 目标与结论

本文面向“算命 / 命理 / 星盘 / 黄历 / 八字”类产品的技术选型，重点分析两个候选 Python 依赖：

- `kerykeion`：西方占星盘、相位、宫位、关系盘、行运盘、SVG 星盘图
- `cnlunar`：中国农历、黄历、节气、干支、八字、宜忌、神煞、时辰等

结论先行：

- 如果项目要做“西方星盘 + 中国黄历/八字”的混合命理系统，`cnlunar` 很适合作为中国传统历法与黄历主引擎。
- `kerykeion` 的功能覆盖很强，但其官方 PyPI/GitHub 页面在 2026-03-21 显示为 `AGPL-3.0`，这对闭源商业项目是实质性许可证风险。
- 对于闭源或 SaaS 商业项目，更稳妥的西占方案通常是：
  - 自建：`pysweph` / `pyswisseph` + 自己封装业务层
  - 或保留 `kerykeion` 仅用于独立开源服务
- 中国历法竞品里，`lunar-python` 和 `sxtwl` 都值得纳入备选：
  - `lunar-python` 更偏高层 API、功能广、接入快
  - `sxtwl` 更偏底层天文历法能力、时间范围更广、适合做高精度底座

推荐落地路线：

- 开源项目路线：`kerykeion` + `cnlunar`
- 闭源商业路线：`pysweph` + `cnlunar`，必要时引入 `sxtwl` 做交叉校验

## 2. `kerykeion` 功能用途分析

## 2.1 定位

`kerykeion` 是一个面向西方占星的 Python 库，核心能力是把出生信息或时间地点信息，转换为结构化占星数据和可视化星盘。

根据其 GitHub/PyPI 页面，当前可覆盖：

- 本命盘（natal chart）
- 合盘 / 关系盘（synastry）
- 组合盘（composite）
- 行运盘（transit）
- 回归盘（solar return / lunar return）
- 相位计算（aspects）
- 宫位与宫头比较
- 元素、模式、权重分布统计
- SVG 星盘图生成
- 面向程序消费的结构化输出

## 2.2 适合的业务场景

`kerykeion` 适合以下产品模块：

- 西方占星排盘
- 星盘关系分析
- 行运提醒与运势解释
- 生成可下载的星盘 SVG / 图片
- LLM 命理解读系统的数据底层
- 占星内容平台的批量排盘任务

## 2.3 输入能力

从公开文档可见，`kerykeion` 支持的输入方式较完整：

- 生日、出生时分
- 经纬度
- 时区字符串
- ISO UTC 时间
- 城市 / 国家信息
- 在线地理编码或离线显式坐标模式
- 视角类型切换，如 geocentric / heliocentric

这意味着它适合做：

- 前端录入出生信息后直接排盘
- 已有 UTC 时间的数据批处理
- 多国家用户的国际化占星服务

## 2.4 输出能力

`kerykeion` 的价值不只是“算出位置”，还包括“可直接给业务消费”的结构化产物：

- 行星位置
- 宫位位置
- 星座落点
- 相位关系
- 宫位覆盖关系
- 元素与模式统计
- 关系评分
- 文本报告模块
- SVG 图形星盘

从工程角度看，这比纯天文计算库更接近产品层能力，能明显减少二次开发成本。

## 2.5 工程优势

- API 已经围绕“数据工厂 + 渲染器”重构，结构比老式单类库更清晰。
- 有现成图形输出，适合快速做 MVP。
- 对本命、合盘、行运、回归盘等场景支持完整。
- 文档、示例、API reference 相对齐全。
- Python 3.9+ 生态友好。

## 2.6 风险与边界

### 许可证风险

这是 `kerykeion` 最大风险点。

2026-03-21 查到的 PyPI 与 GitHub 页面均标示其许可证为 `AGPL-3.0`。这意味着：

- 如果你把它直接集成进闭源项目，合规风险很高。
- 即便是 Web 服务形态，AGPL 也比 GPL 更严格。
- 官方页面同时建议闭源/商业场景考虑其付费 API。

因此：

- 开源项目可优先考虑 `kerykeion`
- 闭源商业项目不建议直接把它作为核心内嵌依赖

### 业务边界

`kerykeion` 更偏西方占星，不覆盖中国传统命理所需的：

- 农历换算
- 八字四柱
- 节气切月
- 黄历宜忌
- 神煞体系
- 传统择日

所以它不能单独支撑“中式算命系统”，必须配合中国历法库。

## 2.7 综合评价

如果只看“西方占星产品能力”，`kerykeion` 很强，属于高层成品化库；如果看“商业落地稳妥性”，许可证是必须单列评审项。

## 3. `cnlunar` 功能用途分析

## 3.1 定位

`cnlunar` 是面向中国农历 / 黄历 / 八字 / 节气 / 神煞体系的高层 Python 库。它不是简单的“农历转换器”，而是把传统命理常用的数据面一次性打包输出。

从其 GitHub 页面可见，核心定位包括：

- 农历计算
- 二十四节气
- 节假日
- 星次
- 每日凶煞
- 每日值神
- 建除十二神
- 每日宜忌
- 彭祖百忌
- 每日五行
- 二十八星宿
- 天干地支
- 生辰八字
- 时辰凶吉

## 3.2 适合的业务场景

`cnlunar` 很适合以下模块：

- 八字排盘
- 黄历查询
- 择日系统
- 每日运势
- 节气与农历日历
- 婚嫁 / 开业 / 搬家等宜忌推荐
- 命理助手或 AI 解释系统的数据底层

## 3.3 核心价值

相比只做农历转换的轻量库，`cnlunar` 的核心价值在于“传统命理业务字段非常多”。它更像一个“命理数据聚合引擎”。

工程收益主要体现在：

- 一次输入日期时间，就能拿到大量传统命理字段
- 减少你自己翻资料、建规则库、补字典表的工作量
- 适合直接给前端、推荐系统或 LLM 使用

## 3.4 算法与数据特点

根据项目说明，`cnlunar` 强调以下点：

- 不依赖数据库
- 依赖少
- 运行快
- 使用香港天文台数据处理节气相关核心数据
- 以《钦定协纪辨方书》为主要传统规则依据
- 支持不同八字月柱算法口径

这类设计对命理业务很有用，因为节气日期、月柱切换、宜忌规则一旦错，后续解释都会偏。

## 3.5 输出能力

`cnlunar` 适合直接向业务输出：

- 农历年月日与闰月信息
- 干支年/月/日/时
- 生肖
- 八字
- 今日节日、农历节日
- 今日节气、下一节气
- 宜忌
- 吉神凶煞
- 值神、十二神、星宿
- 五行、纳音、时辰吉凶

如果你的需求是“做一个可解释的中式命理产品”，`cnlunar` 的单位投入产出比很高。

## 3.6 工程优势

- MIT 许可证，商业集成压力小。
- 单包高层输出丰富，开发效率高。
- 不依赖数据库，部署简单。
- 适合 API 服务化，也适合本地批量计算。

## 3.7 风险与边界

### 口径风险

中国命理系统里，不同流派对以下问题经常有口径差异：

- 立春切年还是春节切年
- 月柱切换口径
- 子时跨日口径
- 真太阳时是否参与计算
- 神煞与宜忌规则采用哪一套古籍体系

`cnlunar` 虽然支持部分切换，但在产品层必须做“统一口径定义”，否则同一用户在不同页面可能看到不同结果。

### 天文底层能力边界

`cnlunar` 输出非常丰富，但它不是像 `sxtwl` 那样偏底层天文历法引擎。若你要做以下场景：

- 超大时间范围历史考据
- 真太阳时深度校正
- 更底层的历法研究
- 与多算法引擎做精度对照

那么应该考虑把 `sxtwl` 作为底层补充。

## 3.8 综合评价

如果产品重点在“黄历、八字、节气、择日、每日宜忌”，`cnlunar` 是非常合适的主引擎，且商业可用性明显好于 AGPL 方案。

## 4. 竞品分析

下面把竞品拆成两组看：西方占星竞品、中国历法竞品。

## 4.1 西方占星竞品

### 竞品 A：`flatlib`

定位：

- 传统西方占星 Python 库

优点：

- 社区中知名度较高
- 适合做传统占星计算
- 结构相对经典，便于理解基础对象模型

不足：

- PyPI 最近一次发布较早，2026-03-21 查询到最新 PyPI 发布为 2021-04-05 的 `0.2.3`
- 可视化、现代 API、产品化输出不如 `kerykeion`
- 更偏计算库，不是完整业务交付层

适用：

- 想要较传统、较轻的西占计算能力
- 愿意自己补图形、报告、业务结构化层

### 竞品 B：`pysweph` / `pyswisseph`

定位：

- Swiss Ephemeris 的 Python 绑定
- 更底层、更高精度的天文/占星计算能力

优点：

- 行业通用底座认知强
- 时间范围广，精度高
- 适合作为严肃占星计算引擎
- 更适合自建中台或底层服务

不足：

- 更底层，业务封装工作量大
- 图表、相位报告、业务模型需要自己搭
- 2026-03-21 查到的 `pysweph` PyPI 页面也标示 `AGPL-3.0`

适用：

- 需要自控算法栈
- 可以接受更高研发投入
- 需要把“计算内核”和“命理解读层”彻底分离

### 西占竞品小结

如果按“上手速度”排：

- `kerykeion` > `flatlib` > `pysweph`

如果按“底层可控性”排：

- `pysweph` > `flatlib` > `kerykeion`

如果按“闭源商业稳妥性”排，需要单独做许可证法务评审，不能只看功能。

## 4.2 中国历法竞品

### 竞品 C：`lunar-python`

定位：

- 高层中国历法 / 老黄历 / 八字工具库

优点：

- MIT 许可证
- 功能覆盖很广
- API 偏高层，接入速度快
- 包含节假日、星座、八字、五行、十神、值星、吉凶等

不足：

- 虽然覆盖很广，但不同细分字段的规则口径仍需逐项验收
- 若你非常看重某一派黄历规则，需要做样本比对

适用：

- 要快速做完整黄历/八字产品
- 想要与 `cnlunar` 并行 POC 比较

### 竞品 D：`sxtwl`

定位：

- 寿星天文历底层引擎，偏天文历法基础设施

优点：

- 支持范围广，GitHub 页面说明 BC722 年以后可与实历相符
- 支持农历、公历、干支、回历、真太阳时、行星历
- 更适合做底层算法依赖或精度校验引擎

不足：

- 使用体验比高层库更偏底层
- 做业务字段拼装时需要更多二次开发

适用：

- 高精度底层历法能力
- 真太阳时相关扩展
- 历法核验

### 竞品 E：`LunarCalendar` / `LunarCalendarX`

定位：

- 轻量农历阳历转换与节日、节气支持

优点：

- 节日、节气、多语言扩展思路清晰
- 对简单日历类产品友好

不足：

- 业务深度不如 `cnlunar` 和 `lunar-python`
- 更像“农历日历工具”，不是“完整命理引擎”

适用：

- 节日/节气日历
- 轻量国际化农历应用

### 中国历法竞品小结

如果按“黄历/命理字段丰富度”排，现阶段可优先比较：

- `cnlunar`
- `lunar-python`
- `sxtwl`（底层核验位）

## 6. 推荐方案文档

## 6.1 方案目标

目标产品：一个可同时支持“西方星盘 + 中国黄历 / 八字 / 节气 / 宜忌”的命理服务系统。

输出形态建议：

- API 服务
- 后台批量任务
- 前端星盘渲染 / 运势页
- LLM 命理解读接口

## 6.2 推荐架构

### 方案 A：开源项目优先

技术栈：

- 西方占星：`kerykeion`
- 中国历法：`cnlunar`
- 可选核验：`sxtwl`

适合场景：

- 开源仓库
- 内部研究工具
- 原型验证
- 允许遵守 AGPL 的项目

优点：

- 交付速度最快
- 星盘图和结构化占星数据现成
- 中式命理字段开箱即用

风险：

- `kerykeion` 的 AGPL 不能忽略

### 方案 B：闭源商业优先

技术栈：

- 西方占星：自建封装层，底层优先评估 `pysweph` / `pyswisseph`
- 中国历法：`cnlunar`
- 精度核验：`sxtwl`

适合场景：

- 闭源商业产品
- SaaS 平台
- 需要长期可控的计算中台

优点：

- 中式命理部分可快速上线
- 西占部分更利于长期掌控数据模型和解释体系

代价：

- 西占的图形、相位解释、关系分析、报告层要自己搭

### 方案 C：双引擎校验方案

技术栈：

- 中国历法主引擎：`cnlunar`
- 中国历法核验引擎：`lunar-python` 或 `sxtwl`
- 西占主引擎：按开源/闭源路线分别选

适合场景：

- 命理结果不能频繁出错
- 希望建立自动化回归样本库
- 需要验证立春切换、节气边界、子时跨日等问题

## 6.3 推荐数据模型

无论用哪套依赖，建议在项目内部统一一层领域模型，不要把第三方库输出直接暴露到业务层。

建议至少拆成以下对象：

- `BirthProfile`
  - 姓名
  - 公历出生时间
  - 时区
  - 经度、纬度
  - 出生地文本
- `WesternChart`
  - 行星位置
  - 宫位
  - 相位
  - 星盘元数据
- `ChineseCalendarProfile`
  - 农历日期
  - 干支四柱
  - 节气信息
  - 生肖
  - 星宿
- `ChineseFortuneProfile`
  - 宜忌
  - 吉神凶煞
  - 值神 / 十二神
  - 时辰吉凶
- `InterpretationContext`
  - 用户语言
  - 流派配置
  - 算法口径配置

这样做的好处：

- 更换底层库时，业务接口不用重写
- 可以同时接多个引擎做 AB 比对
- 有利于给 LLM 提供稳定的 JSON 输入

## 6.4 关键工程策略

### 策略 1：许可证先于编码

在正式上线前，必须先确认：

- 仓库是否开源
- 服务是否闭源
- 是否把第三方库直接集成到主应用
- 是否要拆成独立服务

否则后续重构成本会很高。

### 策略 2：先定义算法口径

建议把以下配置做成系统级常量：

- 年柱切换规则
- 月柱切换规则
- 子时跨日规则
- 是否采用真太阳时
- 宜忌规则来源
- 西占宫位制式
- 相位容许度

### 策略 3：建立样本回归库

建议准备至少 200 组测试样本，覆盖：

- 节气交接当天
- 立春前后
- 闰月
- 子时边界
- 历史日期
- 国内外多时区出生样本

每次升级依赖库时自动比对输出，避免静默漂移。

### 策略 4：让解释层与计算层分离

命理产品里，计算结果只是底层事实，真正的用户体验来自解释层。因此建议：

- 计算层只产出结构化事实
- 解释层单独管理模板、规则、LLM prompt
- 前端图形渲染层也单独隔离

## 6.5 实施步骤

### 第一阶段：2 周内 POC

- 用 `cnlunar` 跑通农历、八字、宜忌、节气 API
- 用 `kerykeion` 或备选西占引擎跑通本命盘与相位 API
- 统一输出 JSON schema
- 做 20 组样本人工验算

### 第二阶段：4 周内中台化

- 封装统一命理领域模型
- 建立测试样本回归
- 建立算法口径配置中心
- 输出 REST / RPC 接口

### 第三阶段：产品化

- 前端星盘图与黄历页整合
- LLM 解读模板上线
- 做缓存、限流、批量任务
- 加入监控与版本追踪

## 7. 最终建议

如果你的目标是“尽快做出一个完整的命理原型”：

- 优先：`cnlunar` + `kerykeion`

如果你的目标是“闭源商业化并长期维护”：

- 优先：`cnlunar`
- 西占部分谨慎评估 `kerykeion`
- 更建议走“底层计算引擎 + 自研业务封装”的路线

如果你的目标是“高精度、可追责、可长期演进”：

- 中国历法主引擎：`cnlunar`
- 中国历法核验引擎：`sxtwl`
- 西占引擎：单独做许可证审查后再定

一句话总结：

- `cnlunar` 更像可直接落地的中式命理业务引擎
- `kerykeion` 更像功能强但许可证需要重点审查的西占高层引擎
