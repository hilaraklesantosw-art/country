# Codex 入门到实践：小白也能做出网页应用

## 前言

这不是一本介绍 AI 概念的书，也不是一本堆参数和术语的工具说明书。

这本书的目标只有一个：让一个几乎不懂 Codex 的初学者，跟着书里的步骤，最终完整做出一个可以运行、可以继续迭代、可以上传到 GitHub 的网页应用。

如果你以前使用过聊天式 AI，但没有真正让 AI 进入项目目录、读代码、改代码、跑验证、提交到仓库，那么你会发现 Codex 和你熟悉的问答式工具很不一样。它不是“帮你出主意”这么简单，而是更像一个在终端里与你协作的工程助手。你给它目标，它会去读仓库、分析结构、修改文件、运行命令、检查结果，再继续把任务推进下去。

但这并不意味着 Codex 是魔法。它不能替你承担所有判断，也不能保证一次就写对所有东西。真正高效的使用方式，不是把它当成全自动机器人，而是学会如何把任务讲清楚，如何在合适的目录里启动它，如何让它在 Git 仓库里安全工作，如何查看它改了什么，如何让它帮你一步一步把一个网页应用做出来。

本书会围绕一个具体项目展开：**一个个人任务管理网页应用**。这个项目足够简单，适合初学者理解；也足够完整，能覆盖网页应用开发中最常见的动作：初始化项目、搭页面、加交互、改样式、修 bug、看 Git 改动、提交到远程仓库，以及最后部署上线。

你不需要一开始就精通前端，也不需要精通命令行。你需要做的，是跟着这本书的节奏，理解 Codex 的工作方式，并且在每一章结束时，完成一个明确的阶段结果。读到最后，你不只是“知道 Codex 是什么”，而是会真正拥有一个自己做出来的网页应用。

## 第 1 章：认识 Codex，建立正确预期

很多人第一次接触 Codex，会把它当成一个“更强的聊天机器人”。这几乎一定会让后面的使用方式变形。因为 Codex 的价值不在于闲聊，而在于进入真实项目环境后，能够围绕目标持续推进任务。

这一章的重点，不是把 Codex 神秘化，而是把它拉回现实。你需要先明白三件事。

第一，Codex 的核心场景不是开放式问答，而是任务执行。你可以让它阅读一个代码库、解释项目结构、修改一个 bug、添加一个功能、运行测试、写文档，甚至帮你创建 skill 或提交代码。它更像一个在终端里工作的协作者，而不是只会输出文字的顾问。

第二，Codex 并不意味着“你什么都不用懂”。你依然需要给出目标、确认方向、判断结果是否符合预期。你并不需要精通所有技术细节，但你至少要有能力判断一件事是否跑偏。对于初学者来说，这反而是好事，因为你可以边做边学，而不是先把所有知识补齐再开始。

第三，Codex 的强大来自它和真实环境的结合。它不是凭空想象一个项目，而是可以在你的目录、仓库、文件和命令行中工作。这也是为什么本书后面会一再强调：你需要学会在正确的项目目录里启动 Codex，学会在 Git 仓库里与它协作，而不是把一切都停留在聊天窗口里。

读完这一章后，你应该形成这样一个判断：Codex 是一个帮你在真实项目中完成任务的工具，而这本书的目标，就是让你用它亲手做出一个网页应用。

本章结束时，你应该得到的结果：

- 知道 Codex 和普通聊天式 AI 的区别
- 知道整本书的项目主线是什么
- 知道自己接下来不是“学概念”，而是“跟着做项目”

## 第 2 章：安装、启动与最常用命令

当你真正开始使用 Codex 时，第一件事不是问它“你能帮我做什么”，而是先把运行环境放对。很多初学者踩的第一个坑，不是不会提需求，而是在错误的目录里启动，或者根本没理解权限模式和参数含义。

这一章只讲最常用、最值得记住的命令，不追求全面，但追求够用。

最基础的启动方式是：

```bash
codex
```

这表示在当前目录启动 Codex。如果你已经进入了某个项目目录，这通常是最直接的用法。

更常见的一种方式是：

```bash
codex .
```

它明确告诉 Codex：就在当前目录工作。对于初学者，这个命令比单独输入 `codex` 更直观，因为它强调了“工作目录”这个概念。你不是在一个抽象空间里和 Codex 说话，而是在一个具体项目里让它干活。

如果你的项目不在当前目录，也可以这样启动：

```bash
codex /path/to/project
```

这在你需要切换仓库、或者从别的位置直接进入某个项目时很方便。

还有一个应该养成的习惯是：

```bash
codex --help
```

它不是为了背参数，而是为了让你知道 Codex 具备哪些启动方式和选项。遇到不确定的命令时，先看帮助信息，比盲猜更可靠。

这一章必须重点解释的，是这个高风险参数：

```bash
codex --dangerously-bypass-approvals-and-sandbox
```

这个参数的意思，不是“让 Codex 更聪明”，而是让它绕过部分审批和沙箱限制，获得更直接的执行能力。它的价值在于：当你在一个完全可控、你非常熟悉的环境中，想加快批量修改、重构或脚本执行速度时，它可能更省事。

但它危险的地方也很明确：如果你在陌生仓库、带有复杂脚本的项目、或者包含敏感配置和密钥的机器上直接使用这个参数，你相当于降低了安全保护。对于初学者，本书的建议非常明确：**先学会默认模式，再理解高权限模式的作用边界，不要一开始就把保护网拆掉。**

这一章还要建立一个关键习惯：**尽量在项目根目录启动 Codex。** 只有这样，它对项目结构的理解才会完整，后续对文件、命令和 Git 状态的判断才更稳。对初学者来说，这个习惯看起来不起眼，但它会直接影响后面每一章的顺畅程度。

本章结束时，你应该得到的结果：

- 会在正确目录启动 Codex
- 知道 `codex`、`codex .`、`codex /path/to/project` 和 `codex --help` 的区别
- 理解 `--dangerously-bypass-approvals-and-sandbox` 的用途和风险

## 第 3 章：用 Codex 搭起第一个网页应用骨架

从这一章开始，真正进入项目实战。我们的目标不是继续讨论 Codex，而是让它开始帮你做事：创建一个最小可运行的网页应用。

这一章的任务非常明确：用 Codex 和你一起搭起个人任务管理网页应用的第一版骨架。这个阶段不追求功能完整，只追求把项目跑起来，让你第一次感受到“从一句需求，到一个可访问页面”的过程。

你可以先告诉 Codex 项目目标，例如：

```text
我要做一个简单的个人任务管理网页应用。先帮我初始化一个前端项目，并搭一个可以运行的基础页面。先不要做复杂功能，先让我能在浏览器里看到首页。
```

这类描述的重点，在于目标具体、范围可控，而且告诉它“先做第一步”。对于初学者，这种分阶段推进特别重要。你不需要一上来就要求它做完整产品，而是先让它帮你完成骨架搭建，再逐步追加功能。

在这一章里，你会学到如何让 Codex：

- 帮你选择一个合适的前端技术栈
- 初始化项目
- 生成基础页面结构
- 运行项目
- 检查目录里出现了哪些文件

更重要的是，你会建立一套非常实用的协作方式：**先让 Codex 解释它准备做什么，再让它动手。** 这对初学者尤其重要，因为这样你不会失去对过程的理解。你会看到项目目录是怎么出现的，页面是怎么跑起来的，哪些文件是入口，哪些文件负责样式和逻辑。

到了这一章结束，你不应该只看到一堆文件，而应该真正拥有一个能在浏览器里打开的网页应用初版。

本章结束时，你应该得到的结果：

- 一个能跑起来的网页应用项目
- 对项目目录结构的初步认识
- 第一次完整体验 Codex 的“分析 -> 修改 -> 验证”工作方式

## 第 4 章：边做边学 Git，让项目改动可控

初学者在使用 Codex 时，最容易忽略的一件事就是 Git。很多人以为 Git 是“以后再学”的内容，但一旦你让 Codex 真正修改项目，Git 就不是可选项，而是你保持安全和可控的基础工具。

这一章不会把 Git 讲成一门完整课程，而是只讲与 Codex 协作最密切的部分。

第一个最重要的命令是：

```bash
git status
```

在让 Codex 动手之前，先看一次 `git status`，你就知道当前仓库是干净的，还是已经有未提交改动。这个习惯非常重要，因为它决定了后面你能否分清哪些改动是自己原来的，哪些是 Codex 新加的。

第二个高频命令是：

```bash
git diff
```

当 Codex 改完代码后，不要急着继续下一步。先看 `git diff`，理解它到底修改了哪些文件、哪些地方。这一步能帮你避免两种典型问题：一是它改多了，动了你本来不想动的部分；二是它虽然说“改好了”，但真正的改动和你的目标并不一致。

接下来是提交：

```bash
git add .
git commit -m "完成任务管理页基础功能"
```

这一步的意义，不只是“保存代码”，更是给当前阶段做一个可回溯的切点。对于初学者来说，这会带来很大的安全感。因为一旦你发现后面的修改跑偏了，至少知道前一个稳定状态在哪里。

如果你已经配置了远程仓库，还会用到：

```bash
git push origin main
```

这一章也会讲一个和 Codex 协作密切相关的原则：**不要随便让它执行破坏性的 Git 命令。** 尤其是像 `git reset --hard`、`git checkout --` 这种操作，在你没有完全确认之前，不应该轻易使用。对初学者来说，这类命令带来的损失往往不是“修一修就好”，而是直接把本地修改清掉。

本章结束时，你应该得到的结果：

- 知道为什么网页项目要放在 Git 仓库里
- 学会用 `git status` 和 `git diff` 查看 Codex 的改动
- 学会做一次基础 commit
- 知道如何避免把改动弄丢

## 第 5 章：用 Codex 完成网页应用的核心功能

如果说前三章是在搭台子，那么这一章就是把网页应用真正做出来。我们的目标是让个人任务管理应用具备“可使用”的核心体验，而不是停留在一个空页面。

你会让 Codex 帮你逐步加入这些功能：

- 新增任务
- 勾选完成任务
- 删除任务
- 显示未完成与已完成状态
- 按状态筛选任务
- 优化页面布局和样式

这一章最重要的方法，不是一次性丢给 Codex 一个巨大需求，而是把它拆成一段一段的任务。例如：

```text
现在在现有页面上加入新增任务功能。用户输入内容后点击按钮，可以把任务加入列表。先完成这个功能，再告诉我你改了哪些文件。
```

接下来再提下一步：

```text
在当前任务列表上增加完成状态切换，并优化一下每个任务项的布局，让它更清晰。
```

这样的协作方式比一句“帮我把整个应用做完”有效得多。因为你始终知道每一步在做什么，Codex 也更容易稳定推进。

这一章还会让你学会一个很关键的习惯：**当功能变多后，不要只看表面效果，要让 Codex 帮你解释关键逻辑。** 比如任务列表状态存在哪里，按钮点击后是如何更新的，筛选逻辑在哪个文件里实现。这样你做出来的不只是“能跑的页面”，而是一个你真正开始理解、以后也敢继续改的项目。

到了这一章结束，你的网页应用应该已经从一个空架子，变成一个有明显交互、有实际用途的网页工具。

本章结束时，你应该得到的结果：

- 一个具备核心任务管理能力的网页应用
- 学会把功能拆成小任务交给 Codex
- 学会让 Codex 在完成功能后解释关键逻辑

## 第 6 章：调试、验证、风险控制与高权限模式

真正做项目时，出错是正常的。页面跑不起来、某个按钮没有反应、样式错位、功能看起来对但逻辑有问题，这些情况都会发生。初学者和熟练使用者之间的区别，往往不在于“会不会犯错”，而在于出了问题后会不会处理。

这一章要建立的不是“零错误”，而是一套稳定的调试习惯。

首先，你要学会让 Codex 不要急着改，而是先分析。例如：

```text
这个页面启动后报错。先帮我定位原因，不要直接大改代码。告诉我最可能的问题点和你建议的修复方案。
```

这类提示能显著降低误改概率。对于初学者来说，先分析后修改，比让 Codex 直接猛改更稳。

其次，你要养成验证的习惯。不是只听它说“已经修好了”，而是让它继续做验证动作，比如运行项目、检查报错、说明结果。如果项目里有测试命令，也可以让它帮你运行并解释结果。

这一章同样会再次讲到高风险参数：

```bash
codex --dangerously-bypass-approvals-and-sandbox
```

到这里，你已经有了足够上下文来理解它为什么危险。它并不是“高手模式”的装饰，而是一个会放松保护边界的参数。如果你只是一个刚开始做网页应用的初学者，绝大多数时候都不需要它。你真正需要的，是学会先分析、先验证、先看改动，再决定下一步。

本章会给出一个实用的判断标准：只有在你已经很熟悉当前项目，知道目录里有什么脚本、知道哪些命令会执行什么、而且明确需要更高自由度时，才去考虑高权限模式。否则，默认模式更适合初学者。

本章结束时，你应该得到的结果：

- 学会让 Codex 先分析再修改
- 学会检查页面错误和功能问题
- 对高权限模式有清晰边界，而不是盲目尝试

## 第 7 章：把网页应用发布出去，形成完整工作流

到这一章，书里的网页应用已经具备基本功能。最后一步，不是停在本地自我欣赏，而是把它整理好、提交好、推送到 GitHub，并形成一套你以后还能重复使用的工作流。

这一章的核心，不是“上线方式有多少种”，而是让你明白一个完整项目交付至少包括什么：

- 代码能运行
- 核心功能可用
- 项目改动已通过 Git 管理
- 代码已经提交到远程仓库
- 你知道后面如何继续迭代

在这一章中，你会让 Codex 帮你做一些收尾动作，例如：

- 检查项目是否还有明显问题
- 优化首页文案或样式细节
- 补一版简短 README
- 确认 Git 改动
- 完成提交并推送

如果你准备继续进阶，还可以在这一章让 Codex 帮你规划部署步骤，把网页应用发布到一个公开可访问的地址上。即便你这一步还没有真正上线，至少也应该把项目整理到“随时可以上线”的状态。

这一章最重要的不是命令，而是心态变化：你已经不再是“试试看 Codex 能做什么”，而是在用一套明确的方式推进一个真实项目。你知道如何开始，知道如何让它帮你搭架子，知道如何逐步加功能，知道如何看 Git 改动，也知道什么时候要谨慎处理权限和风险。对一个原本不懂 Codex 的人来说，这种变化本身就是这本书最有价值的结果。

当你完成这一章时，这本书的目标也就达成了：一个原本不懂 Codex 的初学者，已经可以跟着书里的路径，用 Codex 完整做出一个网页应用。

本章结束时，你应该得到的结果：

- 一个完整可运行的网页应用项目
- 一份可提交、可托管、可继续迭代的代码仓库
- 一套可以重复使用的 Codex 开发工作流

## 结语：从不会用，到会用，再到敢用

很多工具类内容最大的问题，在于读完以后依然不会动手。这本书希望避开这个问题。它不是把 Codex 的所有能力摊开来讲，而是围绕一个网页应用项目，把最必要、最常见、最容易真的用上的部分串起来。

如果你认真跟着做完这 7 章，你得到的不只是一份知识清单，而是一段真实经验：你启动过 Codex，让它进入项目目录，和它一起搭过页面、加过功能、看过 Git 改动、处理过 bug，最后把成果推到了仓库里。对初学者来说，这种真实经验远比背很多概念更有价值。

接下来，你完全可以把这套方法复制到别的项目上。你可以继续做更复杂的网页应用，也可以开始让 Codex 帮你写脚本、整理文档、创建 skill，甚至参与更完整的软件开发流程。

真正重要的，不是把 Codex 当成无所不能的替代者，而是学会如何把它变成一个可靠的协作者。
