# 第 3 章 上手方式与入口选择

很多人第一次用 Codex 时就选错了入口。结果不是工具不好用，而是工作面和任务形态不匹配。

所以真正的“上手”不是先装软件，而是先选工作入口。

## 3.1 App、CLI、IDE、Cloud 分别是什么

可以把这四种入口简单理解为四种工作姿势：

- App：适合需要多任务切换、项目视图、内建 Git 操作和更完整桌面体验的用户
- CLI：适合习惯终端、强调可控性和透明度的开发者
- IDE：适合希望在编辑器上下文里直接协作的人
- Cloud：适合委托式、后台式、并行化任务

它们不是四个完全独立的产品，而是同一套能力的不同工作面。

## 3.2 新手该从哪一个入口开始

如果你是第一次正式使用 Codex，我更推荐两条路径：

- 偏工程型用户：先从 CLI 开始
- 偏编辑器型用户：先从 IDE 扩展开始

原因很现实。

CLI 最容易帮助你看清楚 Codex 在做什么：它读了什么、跑了什么、改了什么、哪里需要审批。这个透明度对建立正确使用心智非常重要。

IDE 则更容易降低心理门槛，因为它直接贴着你熟悉的编辑环境，适合从小范围修改、解释代码和局部任务开始。

App 和 Cloud 更适合作为第二阶段入口：前者适合做多任务和工作区组织，后者适合做委托执行和并行任务管理。

## 3.3 首次安装与登录应该关心什么

安装本身通常不难，真正值得在教程里讲清楚的是以下几个问题：

- 你是用 ChatGPT 账号体系，还是 API key 体系
- 当前入口支持哪些功能，哪些功能受登录方式影响
- 你的工作目录和权限边界在哪里
- 是否启用了搜索、联网或云端能力

很多新手把“能不能登录”当成唯一问题，其实“登录后跑在哪、能做什么、默认权限是什么”更关键。

## 3.4 第一次选择项目时，不要直接上主仓库

新手最容易犯的错误之一，是一上来就把 Codex 连到自己最核心、最复杂、最难验证的仓库上。

更合理的方式是先准备一个低风险项目：

- 文件量不要太大
- 依赖关系尽量清楚
- 本地可运行
- 测试或最起码的验证路径明确

先用它练三件事：

- 让 Codex 解释项目结构
- 让 Codex 做一个很小的功能改动
- 让 Codex 跑一次验证并总结结果

只要这三步跑通，很多抽象概念都会立刻变具体。

## 3.5 新手第一条任务怎么发

真正好的第一条任务，不是“帮我改一下项目”，而是具备四个要素：

- 目标
- 范围
- 限制
- 验收标准

一个可用的模板可以是：

```text
请先阅读这个仓库的结构，重点关注认证相关模块。
不要修改代码，先告诉我：
1. 认证逻辑主要在哪几个文件
2. 登录请求的入口和调用链是什么
3. 如果我要加一个登录失败重试限制，最可能改哪些位置
输出请按“文件 -> 作用 -> 风险点”的格式整理。
```

这类任务有三个好处：

- 风险低
- 能快速观察 Codex 的理解路径
- 能逼你先建立结构化提问习惯

## 3.6 从“问答”切到“执行”的正确顺序

很多失败都发生在跳步。

正确顺序通常是：

1. 先让它分析
2. 再确认思路
3. 再让它修改
4. 再让它验证
5. 最后让它总结变更与风险

这样做的目的，不是拖慢速度，而是降低返工成本。对高风险项目尤其如此。

## 3.7 不同入口适合的典型场景

App 适合：

- 同时跑多个项目线程
- 需要更完整的任务面板和工作区管理
- 想结合 worktrees、内建 Git 和 Automations

CLI 适合：

- 调试、脚本、改配置、查日志
- 希望全程看清命令与环境
- 注重精确控制和可复现操作

IDE 适合：

- 边看代码边提问
- 以文件级和函数级任务为主
- 更在意上下文切换成本

Cloud 适合：

- 后台长任务
- 并行处理
- 需要把执行从本机抽离出来

## 3.8 本章结论

选入口，本质上是在选工作姿势。新手最需要的不是“功能最多的入口”，而是“最容易建立正确习惯的入口”。先从小项目、低风险任务、透明工作面开始，你会更快跨过试用期，进入真正可复用的使用阶段。
