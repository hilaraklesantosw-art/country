# 第 4 章 把 Codex 接进真实开发流程

会不会用 Codex，分水岭不在于你问过多少问题，而在于你能不能让它进入真实开发流程。

所谓“进入流程”，不是让它偶尔帮你写一段函数，而是让它参与一个完整任务的关键环节。

## 4.1 从任务视角而不是功能视角使用 Codex

很多人使用 AI 编程工具时，习惯按功能提问：帮我解释、帮我改、帮我写测试。

更成熟的方式，是按任务阶段组织协作：

- 需求澄清
- 项目阅读
- 修改设计
- 实施变更
- 执行验证
- 总结与复盘

当你按阶段去组织 Codex，它更容易稳定输出，因为每一步的目标和判断标准都更明确。

## 4.2 读项目：先缩小范围，再要结构化输出

陌生项目阅读，是 Codex 最适合承担的起始工作之一。

一个高质量的读项目任务，不应该只让它“介绍一下这个仓库”，而要明确：

- 优先关注哪个业务域
- 不需要看哪些部分
- 输出格式是什么
- 你下一步要做的事情是什么

例如：

```text
请先阅读这个项目中与支付回调有关的代码。
目标不是全面总结，而是为“排查重复回调导致的重复入账”做准备。
请输出：
1. 入口文件
2. 核心调用链
3. 幂等控制位置
4. 你认为最值得先看的三个风险点
```

这样的提问方式能直接服务下一步动作。

## 4.3 改代码：给清楚边界，不要让它猜

让 Codex 改代码时，最重要的不是“描述得多华丽”，而是边界清楚：

- 改什么
- 不改什么
- 优先保持什么不变
- 改完怎么验证

比如你要它修一个 bug，可以把任务写成：

```text
修复用户列表分页切换后筛选条件丢失的问题。
限制：
- 只修改前端代码
- 不改接口协议
- 不引入新依赖
完成后运行现有测试；如果没有覆盖，请补一条最小测试并说明风险。
```

这类任务往往比“修一下分页 bug”有效得多。

## 4.4 跑测试：把验证从“建议”变成“必须步骤”

很多人让 AI 改完代码后，只看表面 diff，不看验证结果。这是非常危险的使用方式。

成熟的流程应该要求 Codex：

- 明确说明准备运行哪些验证命令
- 在命令失败时给出原因分析
- 如果测试不存在，说明缺口
- 在无法验证时明确标注不确定性

验证不是补充动作，而是 AI 协作中最关键的一环。没有验证，所谓“完成”只是表面完成。

## 4.5 做 review：让 Codex 扮演第二双眼睛

除了写和改，Codex 也适合做 review。

但好的 review，不是让它泛泛评价“代码不错”，而是明确审查维度：

- 是否有行为回归风险
- 是否破坏边界条件
- 是否遗漏测试
- 是否引入安全问题
- 是否改动超出目标范围

你甚至可以要求它按“严重级别 -> 文件 -> 具体问题 -> 建议处理方式”的格式输出。这样它就更像真正的工程 review，而不是随口点评。

## 4.6 提交 diff：不要把 diff 当结果，要把它当证据

很多人会盯着 diff 看，好像只要变更看起来合理，事情就结束了。其实 diff 只是证据，不是结论。

你真正该检查的是：

- diff 是否与目标一致
- 有没有超范围改动
- 有没有隐性副作用
- 有没有配套验证结果
- 有没有需要补充文档或注释的地方

Codex 可以帮你生成 diff、解释 diff、甚至总结 diff，但最终负责判断 diff 是否可接受的人仍然应该是你。

## 4.7 一个可复用的最小闭环

如果你想把 Codex 稳定接进开发流程，至少可以从下面这个最小闭环开始：

1. 先分析问题，不急着动手
2. 输出相关文件、调用链和修改计划
3. 确认边界后开始改动
4. 执行测试或替代验证
5. 总结变更、风险和后续建议

这套闭环最大的价值是：它让“AI 帮我做事”变成“AI 在流程中承担明确角色”。

## 4.8 从个人提效走向团队流程

当一个人已经能稳定用这套闭环时，下一步就不是继续堆更多提示词，而是把它抽象成团队约定：

- 哪些任务适合先让 Codex 分析
- 哪些修改必须人工确认后再执行
- 哪些仓库默认不开网络
- 哪些验证步骤必须跑
- 哪些输出格式要统一

这时，Codex 才真正从个人工具变成团队生产力的一部分。
