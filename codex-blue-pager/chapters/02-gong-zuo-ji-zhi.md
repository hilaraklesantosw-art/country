# 第 2 章 Codex 的工作机制

理解 Codex，不能只停留在界面层。你必须知道它为什么会这样工作，为什么有时候特别高效，有时候又会表现失真。只有机制层看清楚了，后面的使用策略才不会流于表面。

## 2.1 Agent 不是“更会聊天”，而是“更会推进任务”

Codex 的核心不是对话框，而是 Agent 工作方式。所谓 Agent，可以简单理解为：模型不只负责输出一句回答，而是可以围绕目标逐步采取动作。

这些动作通常包括：

- 读取项目文件和配置
- 搜索关键代码路径
- 运行命令或工具
- 修改文件
- 根据执行结果调整下一步动作
- 输出总结、风险和后续建议

所以 Codex 的真正单位不是“回答”，而是“任务执行回合”。

## 2.2 本地模式与云端模式

理解 Codex 时，一个必须分清的问题是：它是在你的机器上工作，还是在云端环境里工作。

本地模式的特点：

- 直接在你的工作目录里读写和执行
- 能更贴近真实项目状态
- 更适合快速迭代、调试、联动本地工具
- 风险控制更多依赖本地沙箱与审批策略

云端模式的特点：

- 更适合委托式任务和后台执行
- 便于并行跑多个工作线程
- 更容易做环境隔离和任务队列化
- 对网络、依赖和外部上下文的设计要求更高

这两者不是谁替代谁，而是服务不同的任务形态。高互动、高本地依赖的场景适合本地；批量、并行、后台处理更适合云端。

## 2.3 Sandbox：它能做什么，不能做什么

Codex 之所以能进入真实项目，又不至于完全失控，关键在沙箱。

沙箱的作用，是约束它对文件系统、命令执行和网络的访问范围。你可以把它理解为给代理划定一个可操作的工作区。工作区以内，它可以更流畅地读写与执行；工作区以外，它应该被限制或要求额外审批。

这背后有两个现实原因：

- AI 代理的错误是真实会落地到系统上的
- 软件开发环境往往包含敏感代码、密钥、配置和部署能力

不理解沙箱的人，很容易在“效率”和“安全”之间做出错误权衡。

## 2.4 Approval：为什么需要审批

审批不是给工具添堵，而是给协作设边界。

Codex 在很多环境里支持不同级别的 approval policy。它的意义在于区分：

- 哪些动作可以默认执行
- 哪些动作必须先征求同意
- 哪些失败后才允许升级权限重试

如果没有审批，代理的执行力越强，风险反而越大。真正成熟的使用方式，不是追求“永远不问我”，而是把审批点放在真正有风险的地方，例如：

- 写工作区之外的文件
- 发起网络访问
- 运行可能破坏环境的命令
- 触达生产或共享资源

## 2.5 Web Search：为什么有时需要联网，有时又不该联网

Codex 的一个重要边界，是它面对“项目内上下文”和“外部信息”时的工作方式不同。

项目内问题，通常优先基于本地文件和已有配置解决。外部问题，例如最新文档、第三方 API 变更、价格、公告、漏洞信息，则需要 web search 或网络访问能力。

问题在于：一旦打开网络，效率提升和风险暴露会同时上升。因为网络带来最新信息，也带来 prompt injection、恶意页面、错误指令和不稳定依赖。

所以一个成熟的教程必须讲清：联网是能力，不是默认美德。

## 2.6 AGENTS.md：把隐性经验写成显性规则

AGENTS.md 是 Codex 生态里极其关键的一环。它本质上是在告诉代理：

- 这个仓库是做什么的
- 代码结构应该从哪里看起
- 常用命令有哪些
- 测试怎么跑
- 哪些文件不要动
- 提交前应该满足什么标准

没有 AGENTS.md，很多经验只能存在于人的脑子里。写了 AGENTS.md，就等于把团队对“正确做事方式”的要求写成一层稳定上下文。

对个人开发者来说，AGENTS.md 是提效工具。对团队来说，它更像治理工具。

## 2.7 Skills：把重复工作做成模块

Skills 的价值，不是增加一个炫技功能，而是把重复场景固化下来。

当你发现某件事情已经反复出现，例如：

- 固定格式的代码审查
- 某类页面的 QA 检查
- 部署前的收尾动作
- 文档同步和版本更新

就应该考虑把它抽成 Skill。这样做的好处是：

- 降低每次重新描述任务的成本
- 统一输出风格和检查项
- 便于团队内共享

## 2.8 Subagents：并行不是越多越好

Subagents 听起来很强大，但真正的关键不是“能不能多开几个”，而是“哪些任务值得并行”。

适合 subagents 的通常是：

- 多个相互独立的信息收集任务
- 写入范围清晰、互不冲突的代码改动
- 可并行验证的非阻塞任务

不适合的则是：

- 紧耦合、强依赖前一步结果的任务
- 尚未理清边界的复杂问题
- 需要统一上下文判断的关键路径动作

并行本身不会自动提高质量。错误地并行，只会更快地产生冲突和噪音。

## 2.9 从机制视角理解 Codex

把以上机制合起来看，Codex 不是“一个特别聪明的回答器”，而是一套围绕任务执行构建的系统：

- Agent 负责推进任务
- Sandbox 和 Approval 负责约束风险
- Web Search 负责补充外部信息
- AGENTS.md 负责沉淀规则
- Skills 负责复用流程
- Subagents 负责在合适的时候并行化

理解这套结构之后，你会发现：Codex 的上限，不主要由一句提示词决定，而是由系统约束和工作流设计共同决定。
