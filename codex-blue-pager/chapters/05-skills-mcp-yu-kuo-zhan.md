# 第 5 章 Skills / MCP / 扩展能力

当你只把 Codex 当成一个会回答问题的工具时，每次协作几乎都要从零开始。真正的效率提升，通常出现在你开始把经验沉淀为扩展能力之后。

## 5.1 为什么扩展能力很重要

AI 工具的早期体验往往依赖“惊艳时刻”，但长期价值来自“稳定复用”。如果一个团队每次都要重新解释相同的审查标准、发布流程或 QA 清单，那么效率并不会线性增长。

Skills 和 MCP 的意义，就是让 Codex 从“单次对话工具”变成“可接入、可扩展、可复用的工程系统”。

## 5.2 Skills 是什么

可以把 Skill 理解成“带有明确说明、资源和流程约束的可复用能力包”。

一个 Skill 通常适合封装这类事情：

- 固定场景的检查流程
- 明确步骤的运维或发布动作
- 某类文档或报告生成
- 特定产品或业务域的分析方法

它的核心价值在于：让高频动作不再依赖每个人重新组织语言。

## 5.3 什么时候值得写 Skill

不是所有事情都值得做成 Skill。以下几类最适合：

- 一件事已经重复做了多次
- 输出格式相对固定
- 质量要求需要稳定一致
- 团队里不止一个人会用到
- 单次重新说明的成本已经明显高于维护 Skill 的成本

如果一个流程只会偶尔发生，或者每次都高度变化，那就未必需要 Skill。

## 5.4 Skill 的正确定位

Skill 不应该是“万事通大礼包”，也不应该塞满冗余规则。一个好的 Skill 通常具备三个特征：

- 任务边界清晰
- 输入预期清晰
- 输出结构清晰

也就是说，Skill 不是写得越多越好，而是越聚焦越好。

## 5.5 MCP 是什么

MCP 可以理解为一种让 Codex 访问外部工具、外部服务和附加上下文的桥接方式。

如果说 Skill 更像“把经验打包”，那么 MCP 更像“把能力接进来”。

适合 MCP 的典型场景包括：

- 访问内部文档平台
- 连接 issue、工单、设计稿或知识库系统
- 调用特定服务的只读或受控接口
- 获取任务执行所需的外部上下文

MCP 的价值在于让 Codex 不再只看本地文件，而能在授权范围内接触更完整的业务环境。

## 5.6 什么时候用 MCP，什么时候不要用

很多人一听到 MCP，就想把所有系统都接进去。这通常是过度设计。

适合用 MCP 的情况：

- 没有这些外部上下文，任务很难高质量完成
- 外部信息更新频繁，手工复制粘贴代价很高
- 需要把访问过程规范化和权限化

不适合用 MCP 的情况：

- 只是偶尔查一次信息
- 人工粘贴上下文已经足够
- 接入成本大于实际收益
- 数据权限和安全边界尚未梳理清楚

MCP 不是“越多越先进”，而是“在真正有价值的地方接得刚刚好”。

## 5.7 如何把重复流程产品化

真正的扩展能力建设，可以按这个顺序推进：

1. 先手工跑通流程
2. 把稳定步骤写成文档或模板
3. 反复验证哪些约束是固定的
4. 再抽成 Skill
5. 只有在需要外部连接时再引入 MCP

这一步顺序很重要。很多团队失败，不是因为技术不行，而是过早试图“平台化”，结果把尚未稳定的流程硬编码进系统里。

## 5.8 扩展的真正目标

无论是 Skill 还是 MCP，它们的最终目的都不是让系统看起来更复杂，而是让组织协作更稳定：

- 让新成员更快上手
- 让经验不只停留在口头
- 让常见任务输出更一致
- 让工具能力和团队规范逐步对齐

如果不能带来这些收益，扩展本身就很可能只是额外负担。
