# 第 7 章 安全、边界与成本控制

如果一本 Codex 教程只讲能力，不讲风险，那它大概率只能帮读者产生短期兴奋，不能帮助读者形成长期使用能力。

因为 AI 编程工具一旦进入真实项目，它面对的就不是“回答对不对”的问题，而是“会不会改错、会不会越权、会不会带来隐性成本”的问题。

## 7.1 安全不是最后补丁，而是前置条件

很多团队在试用 Codex 时，最先关注的是效率，最后才想起安全。这种顺序很危险。

更合理的做法是从一开始就定义：

- 哪些仓库允许接入
- 哪些目录允许修改
- 哪些命令允许自动执行
- 哪些场景必须人工审批
- 是否允许联网和外部搜索

这些问题不解决，再强的能力都可能变成组织风险。

## 7.2 敏感仓库与高风险场景

不是所有代码库都适合同一种策略。对不同类型项目，风险完全不同。

高风险项目通常包括：

- 生产环境直接相关仓库
- 含有密钥、客户数据、支付或安全逻辑的系统
- 团队多人共享且改动影响面极大的仓库
- 依赖复杂、验证成本很高的遗留系统

在这些场景下，应该优先提高边界控制强度，而不是追求“全自动”。

## 7.3 Prompt Injection 不只是网页问题

很多人把 Prompt Injection 理解成“打开恶意网页才会中招”。实际上，它也可能来自：

- 不可信的外部文档
- 被污染的 issue 或评论内容
- 第三方仓库内容
- 通过 MCP 引入的外部数据

所以问题不只是“要不要联网”，而是“引入的外部上下文是否可信、是否隔离、是否需要二次判断”。

## 7.4 错误修改风险：最容易被忽略的实际问题

和安全攻击相比，错误修改其实更常见，也更容易被低估。

这类问题包括：

- 看似合理但行为回归
- 修了表面 bug，破坏隐性假设
- 为了通过当前测试，改坏更大范围逻辑
- 输出非常自信，但验证并不充分

所以你不能只防恶意风险，也要防“善意但错误”的风险。对于 AI 代理来说，这往往更高频。

## 7.5 审批和验证是两道不同的防线

很多人误以为“我已经开了审批，就很安全”。其实审批和验证解决的是两个不同问题。

审批解决的是：

- 要不要允许这个动作发生

验证解决的是：

- 动作发生后结果是否正确

这两者缺一不可。只做审批不做验证，会得到“安全地做错事”；只做验证不做审批，则可能先造成不必要的破坏。

## 7.6 成本不只是 token

说到成本，很多人只盯 token 价格。实际上，Codex 的真实使用成本至少包括四层：

- 模型或任务调用成本
- 云端任务和联网资源成本
- 团队调试与验证的人力成本
- 因错误使用带来的返工成本

后两项往往比 token 本身更贵。

## 7.7 如何控制成本

比较实用的成本控制策略包括：

- 按任务分档，不用高规格模型处理所有事情
- 简单任务优先走轻量模式
- 对重复流程做 Skill 化，减少无效重述
- 在高风险任务里要求先分析后修改，减少返工
- 对输出统一要求验证结果，减少“看起来对”的伪完成

成本控制的目标不是一味省钱，而是让每一份支出都更接近有效产出。

## 7.8 边界意识决定长期上限

一个团队最终能把 Codex 用到什么程度，往往不是由“是否愿意尝鲜”决定，而是由边界意识决定。

谁能分清：

- 哪里该自动
- 哪里该审批
- 哪里该联网
- 哪里必须人工兜底

谁就更可能把 Codex 变成长期、稳定、可治理的工程能力。
