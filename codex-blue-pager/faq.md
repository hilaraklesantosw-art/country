# 常见问题 FAQ

## 1. Codex 和 ChatGPT 到底是什么关系

可以把它们理解为同一技术体系下的不同产品形态。ChatGPT 更偏通用对话与知识协作，Codex 更偏工程任务协作。两者都能写代码，但工作位姿和目标并不一样。

## 2. 新手应该先用 App、CLI 还是 IDE

如果你想快速建立“它到底在做什么”的清晰认知，优先用 CLI。如果你更在意低门槛和编辑器内体验，可以先用 IDE。App 和 Cloud 更适合进入第二阶段后再用。

## 3. Codex 会不会替代程序员

短期内它更像能力放大器，而不是责任替代者。它可以承担大量分析、初稿、执行和整理工作，但真正的业务判断、边界取舍和上线责任仍然应该由人承担。

## 4. 要不要一上来就给它开很高权限

不建议。更稳妥的策略是默认保守、按需放开。先让它在低风险项目和低权限环境里跑通，再逐步扩大执行边界。

## 5. Skill 和 MCP 哪个更重要

它们解决的问题不同。Skill 偏流程复用，MCP 偏外部能力接入。对大多数个人开发者来说，先把稳定流程写成模板或 Skill，比着急接一堆 MCP 更重要。

## 6. 模型越强是不是就一定越好

不一定。复杂、高风险任务确实更适合强模型，但结构明确的小任务、重复任务和成本敏感任务，往往更适合更轻量的模型与配置。

## 7. 如果团队还没有规范，能不能先上 Codex

可以试，但要有心理预期：Codex 往往会放大团队现状。规范弱的团队也能从中获得一些效率，但很难在长期形成稳定收益。越早开始写规则、模板和验证清单，越容易把价值留下来。

## 8. 最容易犯的错误是什么

最常见的三个错误是：

- 给模糊任务，却期待高质量结果
- 改完不验证，只看 diff
- 一味追求自动化，却没有先理清边界和风险

## 9. 这本蓝皮书更适合谁

它更适合想建立长期方法的人。如果你只想看几个快捷指令和速成玩法，这本书可能显得偏重；但如果你想认真把 Codex 接进工作流，它会更有价值。
